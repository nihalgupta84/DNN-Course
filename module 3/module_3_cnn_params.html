Parameter Sharing
Why the same kernel weights scan the entire input — and why that's revolutionary
▾

Parameter sharing is arguably the single most important design decision in CNNs. It means that one kernel's weights are reused at every spatial position across the entire input. This single idea reduces a 33-million-parameter layer down to 320 parameters — and often produces a better result.

The Problem It Solves

Imagine training a fully-connected network to recognise a horizontal edge. The network learns weights that fire for edges in the top-left region. Now the same edge appears bottom-right — those are completely different weights, so the network has to re-learn "horizontal edge" for every possible position. A 224×224 image has 50,176 pixels. The network must learn the same concept ~50,000 times over. That's catastrophic.

🔍 Analogy — One Stamp, Whole Document
Think of a rubber stamp shaped like the letter "A". To mark every "A" in a document you press the same stamp at each position — you do not carve a new stamp for each location. That stamp is the kernel. Its shape (the weights) is learned once and applied everywhere.

Concrete Parameter Count

Consider a 32×32 grayscale input processed into 32 output feature maps:

Layer TypeWeights per output unitTotal Parameters
Fully Connected32 × 32 = 1,024 (unique per unit)33,554,432
Conv2d 3×3 (32 filters)3 × 3 = 9 (shared everywhere)320 (9×32 + 32 bias)
✅ 100,000× Reduction
For the same output tensor shape, parameter sharing reduces weights from 33.5 million to just 320 — over 100,000× fewer. And every one of those 9 kernel weights is trained on gradients from all spatial positions, giving a far richer learning signal than any single FC weight that sees only one corner of the image.

The Math Behind It

For a Conv2d layer the output at position \((i,j)\) for output channel \(c_{out}\) is:

Parameter Sharing — same W used at every (i, j)
$$Y[c_{out},\; i,\; j] = \sum_{c_{in}} \sum_{p=0}^{K-1} \sum_{q=0}^{K-1} X[c_{in},\; i{+}p,\; j{+}q] \cdot \underbrace{W[c_{out},\; c_{in},\; p,\; q]}_{\text{identical at every }(i,j)}$$

The indices \(p, q\) scan the kernel (size \(K \times K\)), while \(i, j\) loop over all \(H_{out} \times W_{out}\) spatial positions. The tensor \(W\) appears unchanged in every one of those spatial computations — that is parameter sharing.

Two Key Consequences

🔁
Translation Equivariance
If the input pattern shifts by \((dx, dy)\), the feature map shifts by the same amount. The network's response literally moves with the pattern.
📊
Richer Gradients
Each kernel weight receives gradients from all spatial positions during backprop — not just one region. This makes learning faster and more stable.
🧠
Better Generalisation
A feature detector learned in one corner works everywhere automatically. The network generalises to new positions it never saw during training.
⚡
Hardware Efficiency
Fewer parameters fit in L1/L2 cache. Convolution becomes a single batched matrix multiply — highly optimised on GPUs via cuDNN.
Python · Verifying parameter sharing in PyTorch
import torch
import torch.nn as nn

conv = nn.Conv2d(1, 1, 3, bias=False)

# Kernel has exactly 9 parameters — regardless of image size
print(conv.weight.shape)          # torch.Size([1, 1, 3, 3])
print(conv.weight.numel())         # 9

# Prove same 9 weights handle ANY image size
for size in [32, 64, 224, 512]:
    x = torch.randn(1, 1, size, size)
    y = conv(x)
    print(f"Input {size}×{size} → Output {y.shape[2]}×{y.shape[3]}, params={conv.weight.numel()}")
# Params stay 9 no matter how large the image — parameter sharing!

# FC equivalent for 224×224 → 222×222 would need:
fc_params = (224*224) * (222*222)
print(f"FC equivalent: {fc_params:,} parameters")   # 2,458,813,056

Receptive Field
How much of the original image each neuron "sees" — and how it grows with depth
▾
📖 Definition
The receptive field of a neuron is the region of the original input that influences that neuron's output value. A neuron with a small RF detects fine local detail (edges). A neuron with a large RF can capture global context (objects, scenes). Growing the RF is one of the primary architectural goals in deep CNN design.

Single Layer: Local RF

For a single conv layer with kernel size \(K\), each output neuron directly looks at a \(K \times K\) patch of its input. After one 3×3 conv layer, every output neuron sees exactly a 3×3 region of the original image. That is its receptive field.

How RF Grows With Depth

The power of stacking layers is that the receptive field grows with each layer, even though each individual kernel is small. After two 3×3 conv layers, each neuron in layer 2 looks at a 3×3 patch of layer 1 — but each of those layer-1 neurons already sees a 3×3 patch of the input. So each layer-2 neuron indirectly covers a 5×5 region of the original image. After three layers: 7×7. This growth is the whole point of making networks deep.

Receptive Field Grows Layer by Layer — 3×3 kernels, stride=1

General RF Formula

When layers have different kernel sizes or strides, we compute the receptive field recursively. Start at \(RF_0 = 1\) and apply this update for each layer:

General RF Formula — accounts for stride in earlier layers
$$RF_L = RF_{L-1} + (K_L - 1) \times \prod_{i=1}^{L-1} S_i$$
The product term \(\prod S_i\) is the cumulative stride — each previous stride "stretches" how far a step in the current layer maps back to the input.

Why RF Matters for Network Design

To classify whether an image contains a cat, a neuron must eventually see the whole cat — its RF must be large enough to encompass the object. Too small an RF → the network can only detect textures, not shapes or objects. Too large and it's wasteful. This tension directly informs choices like: how many layers, whether to use pooling (which rapidly grows RF), and whether to use dilated convolutions.

✅ VGGNet's Core Insight (2014)
Stacking small 3×3 convolutions achieves the same receptive field as one large kernel, but with fewer parameters and more non-linearities (one ReLU per layer). VGGNet proved that depth with small kernels consistently beats shallow networks with large kernels — and this is now universally accepted design wisdom.
Python · Computing RF analytically for any architecture
def compute_rf(layers):
    """layers: list of (kernel_size, stride) tuples. Returns receptive field size."""
    rf, stride_product = 1, 1
    for k, s in layers:
        rf += (k - 1) * stride_product
        stride_product *= s
    return rf

# VGG-style: all 3×3 conv, stride=1, but with 2×2 max pool (stride=2) between blocks
# Pool layers also grow RF: treat as (kernel=2, stride=2)
vgg_block1 = [(3,1), (3,1), (2,2)]   # 2 convs + pool
vgg_block2 = [(3,1), (3,1), (2,2)]
vgg_5layers = vgg_block1 + vgg_block2
print(f"VGG after 2 blocks: RF = {compute_rf(vgg_5layers)}")   # RF = 16

# ResNet stem: 7×7 conv stride=2, then 3×3 conv stride=2 max pool
resnet_stem = [(7,2), (3,2)]
print(f"ResNet stem: RF = {compute_rf(resnet_stem)}")             # RF = 11

# Compare: 3 layers of 3×3 stride=1 (no pool)
shallow = [(3,1)] * 3
print(f"3x Conv3×3: RF = {compute_rf(shallow)}")                   # RF = 7

# Stride dramatically grows RF: 3 layers stride=2
strided = [(3,2)] * 3
print(f"3x Conv3×3 stride=2: RF = {compute_rf(strided)}")          # RF = 15

Pooling Layers
Spatial downsampling — reducing resolution while preserving what matters
▾
📖 Definition
A pooling layer reduces the spatial dimensions (H and W) of a feature map by aggregating values in local windows. Unlike convolution, pooling has no learnable parameters — it applies a fixed function (max, average) and intentionally discards fine spatial detail to produce a compact, spatially invariant representation.

Why Do We Need Pooling?

📉
Reduce Computation
Halving H and W cuts the feature map to ¼ its original size — every subsequent conv layer becomes 4× cheaper.
🔭
Grow Receptive Field
After pooling, each spatial position represents a larger region of the original image, giving later layers broader context.
🛡️
Local Invariance
Max pooling is insensitive to small translations within a pool window — if a feature shifts 1px, the max value often stays unchanged.
📦
Hierarchical Features
Progressively smaller feature maps force the network to build increasingly abstract, coarse representations — the semantic hierarchy.

Max Pooling — "Was This Feature Detected?"

Max pooling takes the maximum value in each local window. The intuition is that max pooling is asking a binary-style question: was this feature active anywhere in this neighbourhood? The maximum value captures the strongest activation, telling us that yes, the feature detector fired strongly somewhere in this region, without needing to remember exactly where.

2×2 Max Pooling, Stride=2 — each colour group maps to one output (its max)

Average Pooling — "How Active Was This Region?"

Average pooling computes the mean of each window. Where max pooling asks was the feature present, average pooling asks how active was this region overall. It retains a contribution from every activation — even the weak ones — rather than discarding all but the strongest. This gives smoother downsampling and is often preferred for intermediate pooling in modern architectures, and almost universally used at the end of the network via Global Average Pooling.

Average Pooling on the same input — all four quadrants
$$\text{avg}[0,0]=\frac{1+3+2+4}{4}=2.5 \quad \text{avg}[0,1]=\frac{5+7+6+2}{4}=5.0$$ $$\text{avg}[1,0]=\frac{3+1+8+2}{4}=3.5 \quad \text{avg}[1,1]=\frac{9+4+3+11}{4}=6.75$$

Global Average Pooling (GAP) — Modern Alternative to FC

Global Average Pooling is arguably the most impactful pooling innovation in modern CNNs. Instead of flattening a feature map and feeding it to a huge FC layer, GAP simply averages every spatial position for each channel, producing a single number per channel. The result is a compact vector that can feed directly into a nn.Linear classifier with almost no parameters — and it significantly reduces overfitting.

Global Average Pooling — collapses all spatial positions
$$\text{GAP}(X)[c] = \frac{1}{H \times W}\sum_{i=1}^{H}\sum_{j=1}^{W} X[c,i,j]$$
Input shape (B, C, H, W) → Output shape (B, C, 1, 1) — all spatial info collapsed to one value per channel
TypeOperationLearnable?Used In
MaxPool2dmax in windowNoAlexNet, VGG, LeNet

⚠️ Modern Trend — Replace Pooling with Strided Conv
Many recent architectures replace max pooling with strided convolutions (stride=2). The difference is that strided conv is learnable — the network decides how to downsample rather than always taking the maximum. In practice this often produces better results, especially in generative models and segmentation where the encoder must preserve information carefully.
Python · All pooling variants in PyTorch
import torch, torch.nn as nn

x = torch.randn(1, 32, 64, 64)   # batch=1, 32 channels, 64×64

# Max pooling — halves H and W
print(nn.MaxPool2d(2, 2)(x).shape)              # [1, 32, 32, 32]

# Average pooling — same shape effect, smoother values
print(nn.AvgPool2d(2, 2)(x).shape)              # [1, 32, 32, 32]

# Global Average Pool — collapses all spatial dims
gap = nn.AdaptiveAvgPool2d(1)
print(gap(x).shape)                               # [1, 32, 1, 1]
print(gap(x).flatten(1).shape)                   # [1, 32] — ready for classifier

# Adaptive pool — always produces a fixed output size, regardless of input
print(nn.AdaptiveMaxPool2d((4, 4))(x).shape)     # [1, 32, 4, 4] always

# ── Full ResNet-style classifier head using GAP ──────────────────
class ResNetHead(nn.Module):
    def __init__(self, in_channels=512, num_classes=1000):
        super().__init__()
        self.gap = nn.AdaptiveAvgPool2d(1)      # 512×H×W → 512×1×1
        self.fc  = nn.Linear(in_channels, num_classes)  # 512 params × 1000
    def forward(self, x):
        x = self.gap(x).flatten(1)  # (B, 512)
        return self.fc(x)             # (B, 1000)

# vs. VGG-style FC head: flattened 512×7×7=25088 → 4096 → 4096 → 1000
# ResNet GAP: 512 → 1000 = 512,000 params vs VGG FC: 102+16+4 million params

Stride
How many pixels the kernel jumps between positions — controls output resolution
▾
📖 Definition
Stride \(S\) is the number of pixels the kernel moves between consecutive applications. Stride=1 means dense coverage — the kernel shifts one pixel at a time, maximally overlapping windows. Stride=2 means the kernel skips every other position, producing a smaller output with no overlap between adjacent windows.
Stride=1 vs Stride=2 — how far the 3-element kernel window jumps each step

Output Size Formula

Output dimension with stride S and padding P
$$H_{out} = \left\lfloor \frac{H_{in} + 2P - K}{S} \right\rfloor + 1$$
For K=3, P=1, S=1: \(H_{out} = H_{in}\) (same). For K=3, P=1, S=2: \(H_{out} = \lceil H_{in}/2 \rceil\) (halved).
Input HKPSOutput HNotes
224311224Same padding — keeps resolution
224312112Learnable downsampling
224732112ResNet stem pattern
22430456Aggressive compression
✅ Strided Conv vs Max Pool — Key Trade-off
Both stride=2 conv and 2×2 max pool halve spatial dimensions. But strided conv is learnable — the network optimises the downsampling function for the task. Max pool always takes the maximum, which is a hand-designed choice. For discriminative tasks (classification), both work well. For generative or reconstruction tasks (autoencoders, U-Net), strided conv is strongly preferred because the learned downsampling better preserves information that the decoder needs to reconstruct.
Python · Stride in PyTorch — practical examples
import torch, torch.nn as nn

x = torch.randn(1, 3, 224, 224)

# stride=1: output almost same size (222×222 without padding)
c1 = nn.Conv2d(3, 64, 3, stride=1)
print(c1(x).shape)          # → [1, 64, 222, 222]

# stride=2: output halved — same as max pool but learnable
c2 = nn.Conv2d(3, 64, 3, stride=2, padding=1)
print(c2(x).shape)          # → [1, 64, 112, 112]

# ResNet stem: 7×7 conv, stride=2 — aggressive early compression
resnet_stem = nn.Conv2d(3, 64, 7, stride=2, padding=3)
print(resnet_stem(x).shape)  # → [1, 64, 112, 112]

# Asymmetric stride — useful for non-square inputs (panoramic images, etc.)
c3 = nn.Conv2d(3, 64, 3, stride=(2, 1))
print(c3(x).shape)          # → [1, 64, 111, 222]  — only height halved

Padding
Adding border values to control output size and preserve edge information
▾
📖 Definition
Padding adds extra values around the border of the input before the convolution is applied. Without padding, every conv layer shrinks the spatial dimensions by \(K-1\) pixels (a 3×3 kernel shrinks H by 2, a 5×5 shrinks by 4). In a 50-layer network this would reduce a 224×224 image to nothing. Padding prevents shrinkage and also lets the kernel centre itself on the very edge pixels, giving border regions equal attention to interior regions.

The Border Problem

Without padding, consider what happens to a corner pixel. The kernel can only be centred on positions where all K×K weights land inside the input. A 3×3 kernel centred on position (0,0) would need pixels at row −1, col −1 — which don't exist. So without padding, the kernel never covers corner pixels as a centre point; it only reaches them indirectly. This means corner and edge information contributes less to the output than central information. Padding fixes this by giving the kernel a valid neighbourhood everywhere.

Zero Padding p=1 — one ring of zeros added around a 4×4 input, enabling a 3×3 kernel to centre on every original pixel

⚠️ Zero Padding Artifact in Image Generation
When generating images (GANs, VAEs), zero padding can create a visible border artifact — the network learns to recognise padded zeros as "edge of image" and may produce darker or blurrier borders. Reflect or replicate padding avoids this because the padded values are natural continuations of the image content, not foreign zeros. This is why many image generation architectures use reflect padding.
Python · All padding modes in PyTorch
import torch, torch.nn as nn

x = torch.randn(1, 1, 8, 8)   # 8×8 input

# Valid — output shrinks
c_v = nn.Conv2d(1, 1, 3, padding=0)
print(c_v(x).shape)                            # [1,1,6,6]

# Same — p=1 for K=3, output = input size
c_s = nn.Conv2d(1, 1, 3, padding=1)
print(c_s(x).shape)                            # [1,1,8,8] ✓

# PyTorch shorthand — auto-computes padding='same' (works stride=1 only)
c_auto = nn.Conv2d(1, 1, 3, padding='same')
print(c_auto(x).shape)                         # [1,1,8,8] ✓

# Reflect padding — no zero-border artifact, great for image tasks
c_r = nn.Conv2d(1, 1, 3, padding=1, padding_mode='reflect')
print(c_r(x).shape)                            # [1,1,8,8]

# For p=5 on K=3 (unusual large padding): output BIGGER than input
c_big = nn.Conv2d(1, 1, 3, padding=5)
print(c_big(x).shape)                          # [1,1,16,16] — used in ConvTranspose tricks

# Asymmetric padding via nn.ZeroPad2d(left, right, top, bottom)
pad = nn.ZeroPad2d((2, 2, 1, 1))
print(pad(x).shape)                            # [1,1,10,12]

Dilation (Atrous Convolution)
Expanding the receptive field exponentially without adding parameters or losing resolution
▾
📖 Definition — "à trous" = "with holes"
Dilation (also called atrous convolution) inserts gaps of \(d-1\) zeros between each kernel element before applying it. With dilation factor \(d=1\), the kernel works normally — consecutive elements. With \(d=2\), the kernel samples every other input element, effectively looking at twice the area with the same 9 parameters. With \(d=4\), it samples every 4th element. Crucially, the number of parameters stays exactly the same — only the receptive field grows.

Visualising Dilation

Same 3×3 kernel, three dilation factors — receptive field grows from 3×3 to 7×7 to 13×13

The Key Advantage Over Alternatives

To get a large receptive field without dilation, you have two options: use larger kernels (more parameters, more computation) or add more layers (more memory, more compute). Dilation is a third path — it gets you a large RF with the same 9-parameter 3×3 kernel, at the same output resolution, at roughly the same computational cost. The only thing you "pay" is that the kernel now samples sparsely rather than densely.

Where Dilation Is Used

🖼️
Semantic Segmentation
DeepLab series (v1–v4) uses dilated convolutions throughout the encoder to maintain full resolution while growing the RF to capture whole-object context.
🌊
WaveNet (Audio)
Stacks dilated 1D convolutions with exponentially growing d: 1, 2, 4, 8, 16, 32… to cover thousands of audio timesteps with just a few layers.
🔍
Object Detection
Used in backbone networks to maintain high-resolution feature maps for detecting small objects, which disappear after aggressive downsampling.
📊
Time Series (TCN)
Temporal Convolutional Networks (TCN) use exponential dilation on 1D sequences — doubles d each layer to cover exponentially growing history.
Python · Dilation in PyTorch — Conv2d and Conv1d
import torch, torch.nn as nn

x = torch.randn(1, 1, 32, 32)

# Same 3×3 kernel, three dilation factors — all have 9 params
for d in [1, 2, 4, 8]:
    # padding = d keeps output same size as input
    c = nn.Conv2d(1, 1, 3, padding=d, dilation=d)
    rf = d * (3 - 1) + 1
    print(f"dilation={d}: output={c(x).shape[2]}×{c(x).shape[3]}, RF={rf}×{rf}, params={c.weight.numel()}")
# dilation=1: output=32×32, RF=3×3, params=9
# dilation=2: output=32×32, RF=5×5, params=9
# dilation=4: output=32×32, RF=9×9, params=9
# dilation=8: output=32×32, RF=17×17, params=9

# ── WaveNet-style dilated causal 1D convolutions ─────────────────
class DilatedStack(nn.Module):
    """Exponentially growing dilation — covers 1+(2^n - 1)*2 timesteps after n layers."""
    def __init__(self, channels=64, n_layers=8):
        super().__init__()
        self.layers = nn.ModuleList([
            nn.Conv1d(channels, channels, kernel_size=2,
                       dilation=2**i, padding=2**i)  # causal padding
            for i in range(n_layers)                  # d = 1,2,4,8,16,32,64,128
        ])
    def forward(self, x):
        for layer in self.layers:
            x = x + layer(x)[..., :x.shape[-1]]    # residual + trim to length
        return x

# After 8 layers: RF = 1 + 2*(1+2+4+8+16+32+64+128) = 511 timesteps
# That's 511 samples of context using only 8 layers!

ConvTranspose2d — Transposed Convolution
Learned upsampling — going from small to large feature maps
▾
📖 Definition
A transposed convolution (also called a fractionally strided convolution or incorrectly a deconvolution) is a learned upsampling operation. It maps each input value to a \(K \times K\) patch in the output via a kernel, with overlapping patches being summed. This is the spatial "reverse" of a strided convolution — where stride=2 conv halves resolution, ConvTranspose2d with stride=2 doubles it.

Intuition — Spreading Each Value Into a Patch

Regular convolution is a many-to-one operation at each position: it takes a K×K neighbourhood and produces one output value. Transposed convolution is the opposite — it takes one input value and spreads it into a K×K patch in the output, weighted by the kernel. Where adjacent patches overlap, the contributions are added together. The kernel weights are still learned by gradient descent.

ConvTranspose2d — each input value fans out into a kernel-sized patch, overlapping regions are summed

Output Size Formula

The output size of a transposed convolution is the inverse relationship to a regular convolution with the same hyperparameters:

ConvTranspose2d Output Size
$$H_{out} = (H_{in} - 1) \times S - 2P + K + \text{output\_padding}$$
The output_padding parameter adds extra rows/cols to resolve ambiguity: multiple input sizes can map to the same ConvTranspose2d output, so you sometimes need to specify which one you want.

The Checkerboard Artifact Problem

Transposed convolution is infamous for producing checkerboard artifacts in generated images — a visible grid pattern caused by uneven overlap. This happens when the kernel size is not divisible by the stride: different positions in the output receive contributions from different numbers of input pixels, creating a systematic pattern of stronger and weaker activations.

⚠️ Checkerboard Fix — Two Approaches
Option 1: Use kernel sizes divisible by stride — K=2,S=2 or K=4,S=2 — so every output position receives exactly the same number of input contributions. This mathematically prevents uneven overlap.

Option 2 (preferred in practice): Replace ConvTranspose2d with nn.Upsample(scale_factor=2, mode='bilinear') followed by a regular nn.Conv2d. This separates the "make bigger" step (bilinear interpolation — no learned params) from the "refine features" step (conv — learned). This approach is now used in most state-of-the-art image generation and segmentation architectures.

Python · ConvTranspose2d — all key use cases
import torch, torch.nn as nn, torch.nn.functional as F

# ── Basic upsampling ────────────────────────────────────────────────
x = torch.randn(1, 64, 16, 16)

# Double resolution: H_out = (16−1)*2 + 2 = 32
up1 = nn.ConvTranspose2d(64, 32, kernel_size=2, stride=2)
print(up1(x).shape)        # → [1, 32, 32, 32]

# Same but with K=4 — no checkerboard (K divisible by S)
up2 = nn.ConvTranspose2d(64, 32, kernel_size=4, stride=2, padding=1)
print(up2(x).shape)        # → [1, 32, 32, 32] — same output, no artifact

# ── output_padding: needed when S does not evenly divide size ───────
x7 = torch.randn(1, 64, 7, 7)
# Without output_padding: (7-1)*2 + 2 = 14
print(nn.ConvTranspose2d(64, 32, 2, 2)(x7).shape)              # [1,32,14,14]
# With output_padding=1: get 15×15 if needed
print(nn.ConvTranspose2d(64, 32, 2, 2, output_padding=1)(x7).shape) # [1,32,15,15]

# ── Modern alternative: Upsample + Conv (no checkerboard at all) ────
class UpsampleConv(nn.Module):
    """Preferred approach in modern architectures — separates size from features."""
    def __init__(self, in_ch, out_ch, scale=2):
        super().__init__()
        self.up   = nn.Upsample(scale_factor=scale, mode='bilinear', align_corners=False)
        self.conv = nn.Conv2d(in_ch, out_ch, 3, padding=1)
        self.bn   = nn.BatchNorm2d(out_ch)
    def forward(self, x):
        return F.relu(self.bn(self.conv(self.up(x))))

uc = UpsampleConv(64, 32)
print(uc(x).shape)          # → [1, 32, 32, 32] — same result, smoother output

# ── U-Net decoder block combining skip connections ──────────────────
class UNetDecoder(nn.Module):
    """Typical U-Net decoder: upsample encoder output, concat skip, refine."""
    def __init__(self, in_ch, skip_ch, out_ch):
        super().__init__()
        self.up   = nn.ConvTranspose2d(in_ch, in_ch // 2, kernel_size=2, stride=2)
        self.conv = nn.Sequential(
            nn.Conv2d(in_ch // 2 + skip_ch, out_ch, 3, padding=1),
            nn.BatchNorm2d(out_ch), nn.ReLU(),
            nn.Conv2d(out_ch, out_ch, 3, padding=1),
            nn.BatchNorm2d(out_ch), nn.ReLU()
        )
    def forward(self, x, skip):
        x = self.up(x)
        x = torch.cat([x, skip], dim=1)   # concat along channels
        return self.conv(x)

📌 The Design Checklist
When designing a CNN layer, ask yourself these questions in order:

1. Size: Should this layer keep the resolution (stride=1, padding=K//2) or reduce it (stride=2)?
2. RF: Is the receptive field growing large enough to capture the relevant context? Add dilation or more layers if not.
3. Pooling: Are you using Global Average Pooling at the end instead of flattening into a huge FC?
4. Upsampling: If you need to grow resolution (decoder), use ConvTranspose2d(K=2,S=2) or bilinear+conv to avoid checkerboard artifacts.

How These Hyperparameters Interact

These seven concepts are not independent — they interact to determine the overall information flow through the network. Stride reduces resolution and grows the effective RF. Padding counteracts stride's shrinkage effect. Dilation grows RF without reducing resolution. Pooling reduces resolution aggressively while also growing RF. ConvTranspose2d is the inverse of strided conv. Understanding these interactions lets you read and design network architectures rather than just copying them.
