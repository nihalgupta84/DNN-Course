Vanilla Recurrent Neural Networks

Feedforward networks are stateless — they process each input in isolation, with no memory of what came before. The moment you need to understand a sentence, predict the next word, or classify a heartbeat signal, that assumption breaks down. This page builds the recurrent solution from first principles — and shows exactly why it is not quite enough.

01
Why Sequences? The Fundamental Limitation of Feedforward Networks
What makes sequential data different — and why CNNs and MLPs fundamentally cannot handle it
▾

Everything you have studied so far — MLPs, CNNs, transfer learning — shares one deep assumption that is so obvious it is easy to miss: each input is processed independently. When a CNN classifies an image, it does not care what image came before it or what image will come after. Each image is a standalone problem, and the network is stateless — it has no memory of past inputs.

This assumption works perfectly for static problems. But an enormous class of real-world problems is fundamentally sequential — meaning the correct answer at step t depends not just on the input at step t, but on everything that happened before it. Consider these examples:

💬
Language
The word "bank" means something completely different in "river bank" vs "bank account." You cannot understand word meaning without context — the sequence of surrounding words.
🎵
Audio & Speech
A spoken phoneme like "p" in "pin" vs "spin" is physically different, but your brain uses context to hear the same sound. Speech recognition requires modelling time.
💓
Time Series
Whether a heartbeat is abnormal depends on its relationship to the beats before it. A single data point in isolation tells you almost nothing about arrhythmia.
🎬
Video
A single frame showing a hand might be grasping, releasing, or waving. You can only tell which by watching what happened in the previous frames.
🌡️
Forecasting
Predicting tomorrow's temperature requires yesterday's and today's readings. The trend — not just the current value — is what matters.
🧬
DNA / Protein
The function of a codon depends on its position in a gene. Biological sequences are meaningless without their order.

What Specifically Breaks With a Feedforward Network?

Let's be precise about why an MLP or CNN cannot handle sequences. There are three distinct failures, not just one.

Problem 1 — Fixed Input Size. An MLP requires a fixed-size input vector. But sequences vary in length. A sentence can be 3 words or 3,000 words. You cannot pad everything to the longest possible sequence — that would mean processing thousands of mostly-empty inputs, and the network would have no principled way to distinguish a short sequence from a padded one.

Problem 2 — No Memory Across Steps. Even if you could handle variable length by sliding a fixed window across the sequence, the network would have no memory of what it saw in previous windows. It sees each window in isolation. The network cannot know that the word "not" two sentences ago reverses the sentiment of the current sentence, because that earlier window's information was discarded.

Problem 3 — No Parameter Sharing Across Time. If you tried to handle a length-100 sequence by concatenating all 100 inputs into one giant vector and feeding it to a large MLP, the network would learn separate weights for "the word at position 1", "the word at position 2", etc. The same concept — say, "the subject of the sentence" — appearing at different positions would need to be learned separately for every position. This is both statistically wasteful and structurally wrong.

🧠 Analogy — Reading Without Memory
Imagine reading a book with amnesia — every word you read, you instantly forget. By the time you reach the period at the end of a long sentence, you have no idea what the sentence was about. A feedforward network reading a sequence is exactly this: it processes each element in isolation, with no accumulation of context. The key insight of RNNs is that reading is not a series of independent events — each word updates a running mental model of what you have read so far, and that model persists as you continue reading.

The Key Insight: Hidden State as Memory

The solution that recurrent neural networks introduce is conceptually simple: give the network a hidden state — a vector that persists across time steps. At each step, the network takes two inputs: the current input and the hidden state from the previous step. It produces two outputs: the current output and an updated hidden state that is passed to the next step. The hidden state is the network's memory. It encodes everything the network has seen so far in a compressed, learned representation.

Feedforward vs Recurrent — the fundamental architectural difference

The critical observation is that the same set of weights is reused at every time step. There is only one RNN cell, not a different network for each position. This is analogous to the parameter sharing idea in CNNs — the same filter slides across spatial positions; in an RNN, the same cell slides across time steps. This means the network can generalise across sequence lengths and positions without learning separate parameters for each.

02
The RNN Cell — Architecture and Mathematics
The recurrence equation, weight matrices, and what the hidden state actually means
▾

The RNN cell is the core computational unit, and it is simpler than it first appears. At every time step t, it receives two things: the current input vector xt and the hidden state from the previous step ht−1. It produces two things: the new hidden state ht and optionally an output yt. The same operation, the same weights, repeated at every single time step regardless of sequence length.

The Recurrence Equation

The computation inside the cell is a single equation — a weighted combination of the current input and the previous hidden state, passed through a nonlinearity:

Core RNN recurrence — computed at every time step t
$$\mathbf{h}_t = \tanh\!\left(\mathbf{W}_{hh}\,\mathbf{h}_{t-1} + \mathbf{W}_{xh}\,\mathbf{x}_t + \mathbf{b}_h\right)$$
Output at each step (if required)
$$\mathbf{y}_t = \mathbf{W}_{hy}\,\mathbf{h}_t + \mathbf{b}_y$$

Each symbol has a precise meaning that is worth understanding individually before putting them together.

SymbolShapeRole
xₜ (input_size,) The input at the current time step. For text, this might be a one-hot vector or a word embedding. For audio, a frame of MFCC features.
hₜ₋₁ (hidden_size,) The hidden state from the previous step — the network's memory of everything it has seen so far. At t=0, this is initialised to zeros.
W_xh (hidden_size, input_size) Input weight matrix. Projects the current input into hidden space. Learns "which aspects of the input are worth remembering."
W_hh (hidden_size, hidden_size) Recurrent weight matrix — the memory weight. Projects the previous hidden state. Learns "how to update memory given what was previously remembered." This is the defining matrix of an RNN.
W_hy (output_size, hidden_size) Output weight matrix. Projects the hidden state to the output space. Only used when a prediction is needed at this step.
tanh — Activation function, applied element-wise. Squashes values to (−1, 1), keeping the hidden state bounded. Historically tanh was preferred over ReLU for RNNs because unbounded hidden states cause numerical instability across long sequences.
🔑 W_hh Is What Makes It Recurrent
Remove W_hh from the equation and you have a plain feedforward network applied independently to each time step — exactly what we said cannot model sequences. It is the recurrent weight matrix W_hh that creates the feedback loop, allowing information to flow from one time step to the next. This is the one essential architectural difference between an RNN and everything you have studied before.

Parameter Count — Surprisingly Compact

An RNN has three weight matrices and two bias vectors. If the input has size n and the hidden state has size h, and the output has size o, the total parameter count is:

Total parameters in a vanilla RNN
$$\text{Params} = \underbrace{h \times n}_{\mathbf{W}_{xh}} + \underbrace{h \times h}_{\mathbf{W}_{hh}} + \underbrace{h}_{b_h} + \underbrace{o \times h}_{\mathbf{W}_{hy}} + \underbrace{o}_{b_y}$$

Notice that this count does not grow with the sequence length T. A 256-unit RNN processing sequences of 10 tokens uses the exact same number of parameters as a 256-unit RNN processing sequences of 10,000 tokens — because the weights are shared across all time steps. This is the efficiency advantage of the recurrent approach.

✅ Concrete Example — Character-Level Language Model
Suppose we want to model English text at the character level: 26 letters + punctuation = 65 characters total (input size = 65, one-hot encoded). We choose a hidden size of 128. The RNN has: W_xh = 128×65 = 8,320 params, W_hh = 128×128 = 16,384 params, b_h = 128 params, W_hy = 65×128 = 8,320 params, b_y = 65 params. Total: 33,217 parameters — a tiny model that can be trained on a laptop CPU in minutes and learns to generate plausible-looking text. Andrej Karpathy's famous "The Unreasonable Effectiveness of RNNs" (2015) showed that even this tiny model, trained on Shakespeare, generates surprisingly coherent prose.
03
Unrolling Through Time — The RNN as a Deep Network
How a single recurrent cell becomes a deep computational graph, and what that means for training
▾

The compact diagram of a single RNN cell with a looping arrow is the folded view — convenient for understanding the concept but misleading when you need to think about training. To actually compute gradients, we need to "unroll" the network: replace the single cell with one copy per time step, connected in a chain. This turns the RNN into a deep feedforward network where depth equals sequence length.

This is a critical perspective shift. A sequence of length T, processed by an RNN, creates a computational graph that is T layers deep. Training an RNN on sequences of length 100 is — from the optimiser's perspective — equivalent to training a 100-layer feedforward network. All of the challenges of training very deep networks apply here, magnified by the fact that the same weights appear in every layer (time step).

Folded view → Unrolled view — a T=5 sequence creates a 5-layer computation graph

The Stepped Computation — Following One Sentence

To make this concrete, let's trace what happens when we feed the sentence "The cat sat" into a character-level RNN. At each step, the RNN processes one character and updates its hidden state:

Step 0 / 7
Before processing: h₀ is a zero vector — no memory, no context. The network knows nothing yet. The first character it receives will form the beginning of the context that all subsequent characters build upon.

What Does the Hidden State Actually Encode?

A natural question is: what does the hidden state contain after processing several tokens? The answer is that it contains a learned, compressed representation of the sequence history — but that representation is not interpretable in any human-meaningful way. The hidden state is a dense vector of real numbers whose dimensions do not correspond to named concepts. However, researchers have visualised individual hidden units (single dimensions of the hidden state vector) and found remarkable structure: some units track whether the current character is inside a quotation mark, others track the current indentation level in code, others fire when the previous word was a verb. The network discovers these structured representations without being told to — they emerge from learning to predict the next character.

04
Backpropagation Through Time (BPTT)
How gradients flow backward through the unrolled graph — and why the chain rule becomes dangerous
▾

Training an RNN uses exactly the same algorithm as training any other neural network — gradient descent with backpropagation. The only difference is that backpropagation must flow backwards through the unrolled computation graph, traversing every time step from the last back to the first. This specialised application of backprop is called Backpropagation Through Time (BPTT), though it is really just regular backprop applied to a graph that happens to be unrolled across time.

The Loss Function for a Sequence

If the network makes a prediction at every time step (such as predicting the next character at each position), the total loss is the sum of per-step losses. For a classification task at each step using cross-entropy:

Total loss = sum of per-step losses across all T time steps
$$\mathcal{L} = \sum_{t=1}^{T} \mathcal{L}_t = -\sum_{t=1}^{T} \log P(\mathbf{y}_t \mid \mathbf{x}_1, \ldots, \mathbf{x}_t)$$

The gradient of this total loss with respect to every parameter must be computed. Because every time step uses the same weight matrices (W_hh, W_xh, W_hy), the total gradient for each matrix is the sum of gradients contributed by every time step:

Gradient accumulation — W_hh receives contributions from every step
$$\frac{\partial \mathcal{L}}{\partial \mathbf{W}_{hh}} = \sum_{t=1}^{T} \frac{\partial \mathcal{L}_t}{\partial \mathbf{W}_{hh}}$$

The Chain Rule Unrolled Through Time

The tricky part is computing each ∂Lₜ/∂W_hh. The loss at step t depends on h_t, which depends on h_{t−1}, which depends on h_{t−2}, and so on all the way back to h_0. The chain rule must be applied at every step of this dependency chain. To compute how much the loss at step t depends on the hidden state at step k (where k ≤ t), we must multiply together all the Jacobians of the hidden state at each intermediate step:

Gradient of loss at step t w.r.t. hidden state at earlier step k
$$\frac{\partial \mathcal{L}_t}{\partial \mathbf{h}_k} = \frac{\partial \mathcal{L}_t}{\partial \mathbf{h}_t} \cdot \prod_{j=k+1}^{t} \frac{\partial \mathbf{h}_j}{\partial \mathbf{h}_{j-1}}$$

Each factor in that product is:

Jacobian of one hidden state w.r.t. the previous
$$\frac{\partial \mathbf{h}_j}{\partial \mathbf{h}_{j-1}} = \mathbf{W}_{hh}^\top \cdot \text{diag}\!\left(1 - \mathbf{h}_j^2\right)$$

The diag(1 − h²) term is the derivative of tanh, which equals (1 − tanh²(x)) and lies in the range (0, 1]. Notice what happens when you multiply T of these terms together in the product above: you are multiplying T matrices, each containing factors that are strictly less than 1. This is the seed of the catastrophic vanishing gradient problem, which we will examine in the next section.

BPTT gradient flow — forward pass left→right, backward pass right→left through all T steps

Truncated BPTT — The Practical Compromise

In practice, running full BPTT on very long sequences (thousands of steps) is computationally intractable — the memory required to store all intermediate activations grows linearly with sequence length, and computing gradients through thousands of steps takes too long. The standard solution is Truncated BPTT: process the sequence in fixed-length chunks, propagate gradients only within each chunk, and carry the hidden state forward between chunks. This means the network can maintain state over the full sequence length, but gradients only flow back a limited number of steps. It is a practical compromise that works well in practice even though it theoretically limits what the network can learn from long-range dependencies.

05
The Vanishing & Exploding Gradient Problem
Why vanilla RNNs cannot learn long-range dependencies — the mathematical proof and practical consequences
▾

This is the most important section of Page 1 — it is the problem that motivated every major advance in sequential modelling that came after vanilla RNNs. Understanding it deeply means understanding why LSTMs, GRUs, and ultimately Transformers were invented. It is not merely a training inconvenience — it is a fundamental architectural failure.

The Mathematical Root Cause

Recall from BPTT that the gradient of the loss at time step t with respect to the hidden state at an earlier time step k requires multiplying together (t − k) Jacobian matrices, each of the form:

Gradient product across (t − k) steps
$$\frac{\partial \mathbf{h}_t}{\partial \mathbf{h}_k} = \prod_{j=k+1}^{t} \mathbf{W}_{hh}^\top \cdot \text{diag}\!\left(1 - \mathbf{h}_j^2\right)$$

Consider what happens to this product as (t − k) grows large. Each factor contains two components: W_hh and the tanh derivative diag(1 − h²). Think about the eigenvalues of this product. If the largest eigenvalue of W_hh (the spectral radius) is less than 1, then this product shrinks toward zero exponentially fast. If it is greater than 1, the product grows toward infinity exponentially fast. In both cases, the gradient becomes useless.

Gradient norm grows or shrinks exponentially with distance (t − k)
$$\left\|\frac{\partial \mathbf{h}_t}{\partial \mathbf{h}_k}\right\| \leq \left(\lambda_{\max} \cdot \max_j\left(1 - h_j^2\right)\right)^{t-k}$$
where λ_max is the largest singular value of W_hh. If this base is <1 → vanishing. If >1 → exploding.

The tanh derivative (1 − h²) makes the situation worse: it has a maximum value of 1 (when h = 0) and approaches 0 as the hidden state saturates near ±1. Saturated hidden states push the Jacobian eigenvalues toward zero even more aggressively.

Intuition: What This Means in Practice

When the gradient vanishes, it means that the network's weights receive almost zero update signal from time steps that are far in the past. The network effectively has a short memory — it can only "see" a few recent time steps in its gradient signal. It learns to model short-range patterns but is completely blind to long-range dependencies.

To see why this is catastrophic, consider the sentence: "The trophy didn't fit in the suitcase because it was too large." What does "it" refer to — the trophy or the suitcase? The correct answer ("the trophy") requires connecting the word "it" (near the end) with information about the relative sizes of both objects (at the beginning). In a vanilla RNN, the gradient signal from predicting "trophy" must flow back 10+ steps to update the weights that process "trophy" at the beginning. With vanishing gradients, that signal is effectively zero — the network cannot make this connection.

Gradient magnitude reaching step 1 from step T — exponential decay with distance

Exploding Gradients — The Other Problem

When eigenvalues are large (>1), gradients grow exponentially with sequence length instead of shrinking. This causes the weights to receive enormous updates that shoot past any good solution — training becomes violently unstable, loss oscillates wildly, and weights grow until they overflow to NaN. Exploding gradients are actually easier to detect and fix than vanishing gradients: the standard remedy is gradient clipping, where you cap the gradient norm to a maximum value (typically 1.0 or 5.0) before the parameter update. This prevents any single backward pass from making a catastrophically large update.

Python · Gradient clipping — the standard fix for exploding gradients
# After loss.backward(), before optimizer.step():

# This clips the L2 norm of all gradients to max_norm.
# If ||grad|| > max_norm, scale all gradients by max_norm/||grad||
# If ||grad|| <= max_norm, leave gradients unchanged.
torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=5.0)

optimizer.step()

# You can monitor gradient norms before clipping to detect problems:
total_norm = 0
for p in model.parameters():
    if p.grad is not None:
        total_norm += p.grad.data.norm(2).item() ** 2
total_norm = total_norm ** 0.5
# If total_norm >> max_norm consistently, you have exploding gradients

🔴 The Core Failure — Long-Range Dependencies
A vanilla RNN trained with gradient descent has an effective memory of roughly 5–10 time steps in practice. Any information from further back than that contributes essentially zero gradient signal. This means vanilla RNNs:

Can learn: local character patterns, short-range grammar (subject–verb agreement within a clause), n-gram statistics, short-term signal patterns in time series.

Cannot learn: long-range coreference ("it" referring to a noun introduced 20 words earlier), document-level structure, long musical phrases, dependencies in DNA sequences spanning hundreds of bases.
⟶ This Is Why LSTM Was Invented
Hochreiter & Schmidhuber (1997) proved mathematically that vanilla RNNs cannot maintain information across more than roughly 10 time steps due to vanishing gradients. Their solution — the Long Short-Term Memory (LSTM) network — introduces a fundamentally different information pathway: a cell state that flows through the network with only additive (not multiplicative) updates, keeping the gradient highway clear over hundreds of time steps. This is exactly what Page 2 covers.

RNN Architectures — Many-to-Many, Many-to-One, One-to-Many
How input–output structure determines the network topology for different real tasks
▾

One of the most powerful aspects of RNNs is their flexibility: unlike a feedforward network, which has a fixed input size and a fixed output size, an RNN can be configured to accept sequences and produce sequences of completely different lengths — including producing a single output from a long sequence, or producing a long sequence from a single input. The right configuration depends entirely on the task. Understanding these patterns is essential because they directly determine how you build the network and where you compute the loss.

Five canonical RNN input–output configurations — rectangles are vectors, arrows are connections
ConfigurationInput → OutputLoss Computed AtReal Task Examples
Many-to-Many (aligned) Sequence → same-length sequence Every time step Part-of-speech tagging, Named Entity Recognition, video frame labelling
Many-to-One Sequence → single vector Final time step only Sentiment analysis, document classification, music/video genre
One-to-Many Single vector → sequence Every output step Image captioning (CNN encodes image, RNN generates caption), music generation
Many-to-Many (seq2seq) Sequence → different-length sequence Every decoder step Machine translation, speech recognition, summarisation, dialogue

Stacked (Deep) RNNs

Just as deep CNNs outperform shallow ones by learning hierarchical features, stacking multiple RNN layers on top of each other often dramatically improves performance. In a stacked RNN, the hidden state sequence produced by layer 1 becomes the input sequence for layer 2. Each layer learns progressively more abstract temporal representations: lower layers learn low-level patterns (character n-grams, short-range syntax), higher layers learn higher-level abstractions (phrase-level meaning, discourse structure). In PyTorch this is simply nn.RNN(num_layers=2).

Bidirectional RNNs

A standard RNN processes the sequence from left to right, so the hidden state at position t only contains information from positions 1 through t. For many tasks this is a significant limitation — to predict the label for the word "bank" in a sentence, you want to look at words both before and after it. A Bidirectional RNN solves this by running two separate RNNs over the same sequence: one left-to-right (as normal) and one right-to-left. At each position, the hidden states from both directions are concatenated, giving the network access to full context in both directions. The trade-off is that bidirectional RNNs cannot be used for tasks requiring real-time left-to-right generation (like language modelling), but they are ideal for tasks like Named Entity Recognition where the full sequence is available at inference time.

07
PyTorch Implementation — From Scratch and with nn.RNN
Building a vanilla RNN two ways: manually to understand every detail, then cleanly for real use
▾

Understanding the mathematics is one thing — seeing how every tensor flows through PyTorch's computational graph is what solidifies the understanding. We build the RNN two ways: first entirely from scratch using raw nn.Linear layers and a manual loop (so you can see every weight matrix and every hidden state update explicitly), then using PyTorch's built-in nn.RNN (which is what you use in practice).

Part 1 — RNN Built Entirely from Scratch

Python · Vanilla RNN from scratch — every detail exposed
import torch
import torch.nn as nn

class VanillaRNNCell(nn.Module):
    """A single RNN cell — implements: h_t = tanh(W_xh @ x_t + W_hh @ h_{t-1} + b_h)
    
    This is exactly the recurrence equation. By seeing each weight matrix
    separately, you can understand what each one does:
      W_xh: projects current input → hidden space (what's relevant now?)  
      W_hh: projects previous hidden state → hidden space (what to remember?)
      b_h:  bias for the hidden state update
    """
    def __init__(self, input_size: int, hidden_size: int):
        super().__init__()
        self.input_size  = input_size
        self.hidden_size = hidden_size

        # W_xh: maps input to hidden space  shape: (hidden_size, input_size)
        self.W_xh = nn.Linear(input_size,  hidden_size, bias=False)
        # W_hh: maps prev hidden to hidden  shape: (hidden_size, hidden_size)
        self.W_hh = nn.Linear(hidden_size, hidden_size, bias=True)
        # bias b_h is inside W_hh (bias=True above) — PyTorch convention

    def forward(self, x_t, h_prev):
        """
        Args:
          x_t:    (batch, input_size)   — current input at this time step
          h_prev: (batch, hidden_size)  — hidden state from previous step
        Returns:
          h_t:    (batch, hidden_size)  — new hidden state
        """
        # Core equation: h_t = tanh(W_xh @ x_t + W_hh @ h_{t-1} + b_h)
        h_t = torch.tanh(self.W_xh(x_t) + self.W_hh(h_prev))
        return h_t


class VanillaRNN(nn.Module):
    """Full RNN that processes an entire sequence using VanillaRNNCell.
    
    Demonstrates the explicit time-step loop — this is what nn.RNN
    does internally, but hidden from you.
    """
    def __init__(self, input_size, hidden_size, output_size):
        super().__init__()
        self.hidden_size = hidden_size
        self.cell    = VanillaRNNCell(input_size, hidden_size)
        # W_hy: maps final hidden state → output predictions
        self.W_hy    = nn.Linear(hidden_size, output_size)

    def forward(self, x, h0=None):
        """
        Args:
          x:  (batch, seq_len, input_size) — full input sequence
          h0: (batch, hidden_size) or None — initial hidden state
        Returns:
          outputs:    (batch, seq_len, output_size) — prediction at each step
          h_all:      (batch, seq_len, hidden_size) — all hidden states
          h_final:    (batch, hidden_size)           — final hidden state
        """
        batch, seq_len, _ = x.shape

        # Initialise h_0 = zeros if not provided — "the network knows nothing yet"
        if h0 is None:
            h0 = torch.zeros(batch, self.hidden_size, device=x.device)

        h = h0
        outputs = []
        h_all   = []

        # ── The explicit time loop ────────────────────────────────────────
        # This is the heart of an RNN: step through each position in the
        # sequence, updating the hidden state at each step.
        # x[:, t, :] extracts one time step across the whole batch.
        for t in range(seq_len):
            h = self.cell.forward(x[:, t, :], h)  # (batch, hidden_size)
            h_all.append(h)

            # Compute output (prediction) at this step
            out_t = self.W_hy(h)                   # (batch, output_size)
            outputs.append(out_t)

        # Stack list of tensors → (batch, seq_len, *)
        outputs = torch.stack(outputs, dim=1)   # (batch, seq_len, output_size)
        h_all   = torch.stack(h_all,   dim=1)   # (batch, seq_len, hidden_size)
        return outputs, h_all, h


# ── Quick test to verify shapes ───────────────────────────────────
batch_size  = 4
seq_len     = 10
input_size  = 8    # e.g. 8-dim word embedding
hidden_size = 16
output_size = 5    # e.g. 5-class classification at each step

Part 2 — Using nn.RNN (Production Code)

Python · PyTorch nn.RNN — understanding the API and tensor shapes
import torch
import torch.nn as nn

# ── nn.RNN parameters ─────────────────────────────────────────────
rnn = nn.RNN(
    input_size  = 8,     # size of each input vector x_t
    hidden_size = 16,    # size of hidden state h_t (your choice — the key hyperparameter)
    num_layers  = 2,     # stacked RNN: layer2 input = layer1 hidden states
    nonlinearity= 'tanh',# 'tanh' (default) or 'relu'
    bias        = True,  # include bias terms
    batch_first = True,  # ← IMPORTANT: if True, input is (batch, seq, features)
                          #             if False (default!), input is (seq, batch, features)
                          #   Always use batch_first=True to avoid shape confusion
    dropout     = 0.0,   # dropout between stacked layers (not applied to last layer)
    bidirectional=False, # if True, runs forward + backward and concatenates
)

# ── Input and output shapes ───────────────────────────────────────
# With batch_first=True:
batch_size = 4
seq_len    = 10
x = torch.randn(batch_size, seq_len, 8)   # (batch, seq, input_size)

# h0: initial hidden state — (num_layers, batch, hidden_size)
# If not provided, defaults to zeros
h0 = torch.zeros(2, batch_size, 16)   # (num_layers=2, batch, hidden)

out, h_n = rnn.forward(x, h0)

# out:  (batch, seq_len, hidden_size)  — hidden state at EVERY time step
#         useful for: sequence labelling (one output per input step)
# h_n:  (num_layers, batch, hidden_size) — final hidden state of EACH layer
#         useful for: sequence classification (use h_n[-1] as the summary)

print(f"out shape: {out.shape}")    # → (4, 10, 16)
print(f"h_n shape: {h_n.shape}")    # → (2, 4, 16)


# ── Full sequence classifier: Many-to-One ─────────────────────────
# Task: classify a whole sentence into one of N categories (e.g. sentiment)
class SequenceClassifier(nn.Module):
    def __init__(self, vocab_size, embed_dim, hidden_size,
                 num_layers, num_classes, dropout=0.3):
        super().__init__()
        # Embedding layer: converts token indices → dense vectors
        # This is learned jointly with the RNN — no pretrained embeddings needed
        self.embed = nn.Embedding(vocab_size, embed_dim, padding_idx=0)
        self.rnn   = nn.RNN(
            embed_dim, hidden_size,
            num_layers=num_layers,
            batch_first=True,
            dropout=dropout if num_layers > 1 else 0.0,
        )
        self.drop      = nn.Dropout(dropout)
        self.classifier= nn.Linear(hidden_size, num_classes)

    def forward(self, token_ids):
        """
        token_ids: (batch, seq_len)  — integer token indices
        Returns:   (batch, num_classes) — class logits
        """
        x   = self.embed(token_ids)           # (batch, seq, embed_dim)
        _out, h_n = self.rnn(x)               # h_n: (num_layers, batch, hidden)
        # Take the final layer's final hidden state as the sequence summary
        h_last = h_n[-1]                      # (batch, hidden_size)
        h_last = self.drop(h_last)
        return self.classifier(h_last)         # (batch, num_classes)


# ── Sequence tagger: Many-to-Many (aligned) ───────────────────────
# Task: label each token (e.g. POS tagging, Named Entity Recognition)
class SequenceTagger(nn.Module):
    def __init__(self, vocab_size, embed_dim, hidden_size,
                 num_layers, num_tags, dropout=0.3):
        super().__init__()
        self.embed = nn.Embedding(vocab_size, embed_dim, padding_idx=0)
        # Bidirectional: each position sees context from BOTH directions
        # → hidden_size effectively doubles because forward + backward concat
        self.rnn   = nn.RNN(
            embed_dim, hidden_size,
            num_layers=num_layers,
            batch_first=True,
            dropout=dropout if num_layers > 1 else 0.0,
            bidirectional=True,  # ← use both directions for labelling tasks
        )
        # Output size is hidden_size * 2 because bidirectional concatenates
        self.drop   = nn.Dropout(dropout)
        self.tagger = nn.Linear(hidden_size * 2, num_tags)

    def forward(self, token_ids):
        x   = self.embed(token_ids)            # (batch, seq, embed)
        out, _ = self.rnn(x)                   # out: (batch, seq, hidden*2)
        out = self.drop(out)
        return self.tagger(out)               # (batch, seq, num_tags)

The Training Loop — Including Gradient Clipping

Python · Complete training loop for a sequence classifier
model     = SequenceClassifier(vocab_size=10000, embed_dim=64,
                                hidden_size=128, num_layers=2, num_classes=3)
optimizer = torch.optim.Adam(model.parameters(), lr=1e-3)
criterion = nn.CrossEntropyLoss()

for epoch in range(num_epochs):
    model.train()
    for token_ids, labels in train_loader:
        optimizer.zero_grad()

        logits = model(token_ids)          # (batch, num_classes)
        loss   = criterion(logits, labels) # scalar
        loss.backward()                    # BPTT happens here automatically

        # Clip gradients — essential for RNNs to prevent exploding gradients
        nn.utils.clip_grad_norm_(model.parameters(), max_norm=5.0)

        optimizer.step()
🔑 Always Use batch_first=True and Always Clip Gradients
PyTorch's default for nn.RNN is batch_first=False, meaning the input shape is (seq_len, batch, features). This trips up almost every beginner because it is the opposite of every other PyTorch layer (which uses batch as the first dimension). Always pass batch_first=True unless you have a specific reason not to. And always call clip_grad_norm_ before optimizer.step() — without it, vanilla RNNs will frequently diverge on longer sequences.

Application 1 — Character-Level Language Modelling

The simplest and most illuminating application of vanilla RNNs is the character-level language model: given a sequence of characters so far, predict the next character. Despite its simplicity, this task forces the network to learn spelling, grammar, punctuation, code structure — anything that is predictable from context. Karpathy's 2015 demonstration that a small RNN trained on Shakespeare could generate plausible-sounding Shakespeare, and trained on Linux source code could generate compilable-looking C, remains one of the most striking demonstrations that sequence modelling can discover genuine linguistic structure without any linguistic supervision.

Python · Character-level language model — training and generation
import torch, torch.nn as nn

class CharLM(nn.Module):
    """Character-level language model.
    Trained to predict: given characters so far → what is the next character?
    At each time step t, input  = one-hot(char_t)
                          output = predicted distribution over all chars
    Loss = cross-entropy between prediction and true next character.
    """
    def __init__(self, vocab_size, hidden_size, num_layers=1):
        super().__init__()
        self.vocab_size  = vocab_size
        self.hidden_size = hidden_size
        # No embedding layer — we use one-hot vectors directly as input
        # (for small vocabs like 65 chars, one-hot is fine)
        self.rnn = nn.RNN(vocab_size, hidden_size,
                           num_layers=num_layers, batch_first=True)
        self.head = nn.Linear(hidden_size, vocab_size)

    def forward(self, x, h0=None):
        """x: (batch, seq_len) — integer character indices"""
        # One-hot encode: (batch, seq_len) → (batch, seq_len, vocab_size)
        x_onehot = nn.functional.one_hot(x, self.vocab_size).float()
        out, h_n = self.rnn(x_onehot, h0)   # out: (batch, seq, hidden)
        logits = self.head(out)              # (batch, seq, vocab_size)
        return logits, h_n

    def generate(self, seed_char_idx, length, temperature=1.0, device='cpu'):
        """Generate text by repeatedly sampling from the predicted distribution.
        
        temperature > 1: more random / creative
        temperature < 1: more conservative / repetitive
        temperature = 1: sample from the raw predicted distribution
        """
        self.eval()
        generated = [seed_char_idx]
        h = None   # start with zero hidden state

        with torch.no_grad():
            for _ in range(length):
                # Feed only the last generated character at each step
                x = torch.tensor([[generated[-1]]], device=device)
                logits, h = self.forward(x, h)  # h carries context forward

                # Apply temperature: divide logits before softmax
                # Low T → sharper distribution → more predictable output
                # High T → flatter distribution → more surprising output
                probs = torch.softmax(logits[0, 0] / temperature, dim=-1)
                next_char = torch.multinomial(probs, 1).item()
                generated.append(next_char)

        return generated


# ── Training ──────────────────────────────────────────────────────
# Build character vocabulary from text
text    = open('shakespeare.txt').read()
chars   = sorted(set(text))
stoi    = {c: i for i, c in enumerate(chars)}  # char → int
itos    = {i: c for c, i in stoi.items()}       # int → char
data    = torch.tensor([stoi[c] for c in text], dtype=torch.long)

model     = CharLM(vocab_size=len(chars), hidden_size=128, num_layers=2)
optimizer = torch.optim.Adam(model.parameters(), lr=1e-3)
criterion = nn.CrossEntropyLoss()

# Input = chars[0..T-1], Target = chars[1..T] (next char prediction)
chunk_size = 100   # truncated BPTT chunk length
for step in range(10000):
    idx    = torch.randint(len(data) - chunk_size - 1, (32,))
    x_seq  = torch.stack([data[i  : i+chunk_size  ] for i in idx])
    y_seq  = torch.stack([data[i+1: i+chunk_size+1] for i in idx])

    optimizer.zero_grad()
    logits, _ = model(x_seq)                    # (32, 100, vocab)
    loss = criterion(logits.reshape(-1, len(chars)),
                      y_seq.reshape(-1))          # flatten for loss
    loss.backward()
    nn.utils.clip_grad_norm_(model.parameters(), 5.0)
    optimizer.step()

Application 2 — Time Series Forecasting

RNNs are a natural fit for time series: each time step's input is the current measurement (or a window of measurements), and the network learns temporal patterns to predict future values. The hidden state accumulates a learned summary of recent history that informs the prediction. This is used in stock price modelling, weather forecasting, IoT sensor anomaly detection, and energy consumption prediction.

Python · Time series forecaster — predict next N values from a sequence
class TimeSeriesRNN(nn.Module):
    """Predict the next `horizon` values from a sequence of `seq_len` steps.
    
    Each time step input is a vector of features (e.g. [temperature, humidity])
    or just a single scalar (input_size=1 for univariate series).
    """
    def __init__(self, input_size, hidden_size, num_layers, horizon):
        super().__init__()
        self.rnn  = nn.RNN(input_size, hidden_size,
                            num_layers=num_layers, batch_first=True, dropout=0.2)
        # Regression head: no activation — we predict real values
        self.head = nn.Linear(hidden_size, horizon)

    def forward(self, x):
        """x: (batch, seq_len, input_size)"""
        _, h_n = self.rnn(x)         # h_n: (num_layers, batch, hidden)
        h_last = h_n[-1]              # last layer's final hidden state
        return self.head(h_last)      # (batch, horizon) — future predictions

criterion = nn.MSELoss()

Limitations of Vanilla RNNs — and What Comes Next
A complete honest assessment, and why every one of these limitations has a specific architectural solution
▾

Vanilla RNNs were a breakthrough when they were introduced, and they are still worth studying deeply — because every limitation they have directly motivated a specific architectural innovation. Understanding what breaks and why it breaks is the foundation for understanding every technique that comes after them. Let's be precise about the failure modes.

Limitation Why It Happens Real Consequence Solution (coming next)
Cannot learn long-range dependencies Vanishing gradients: multiplying many Jacobians (each with eigenvalues < 1) makes the gradient exponentially small for steps far in the past. The signal from an important word 50 positions back contributes essentially zero gradient to the weights that process it. In practice, effective memory is ≈ 5–10 steps. Cannot resolve coreference, cannot model document structure, cannot capture long musical phrases. LSTM / GRU — gated cell state acts as a gradient highway with additive (not multiplicative) updates.
Sequential computation — cannot parallelise Step t+1 cannot begin until step t is complete, because h_t is required to compute h_{t+1}. This is structural: the temporal dependency chain creates a sequential bottleneck regardless of how much hardware you have. Training on long sequences (e.g. documents, audio) is slow. Cannot use modern massively parallel GPU compute efficiently. A sequence of 1000 steps requires 1000 sequential operations. Transformers — self-attention has no sequential dependency: all positions attend to all others simultaneously, enabling full GPU parallelisation.
Fixed-size hidden state bottleneck The entire history of a long sequence must be compressed into a single h_T vector of fixed dimensionality. Compressing a 1000-word document into 256 numbers is inherently lossy — important details from early in the sequence are overwritten by later information. Encoder-decoder seq2seq models degrade severely on long input sequences: the context vector cannot hold enough information about the beginning of a long sentence to translate the end correctly. Attention mechanism — decoder can directly access all encoder hidden states instead of relying on a single bottleneck vector.
No global context at early steps A left-to-right RNN processes x_1 with only h_0 (zeros) as context. The representation of x_1 is formed before the network has seen any of x_2, x_3, ... x_T — even if later tokens are crucial for understanding x_1. For tasks like NER and POS tagging, the label for a word depends on words that come after it. Unidirectional RNNs miss this. Bidirectional RNNs help but add complexity and preclude autoregressive generation. Bidirectional RNN (partial fix), then BERT-style Transformers (complete fix — every position attends to every other position simultaneously).
Difficult to train reliably Vanishing and exploding gradients make the loss landscape steep and ill-conditioned. Gradient clipping helps with explosions but does not solve vanishing. Learning rates must be carefully tuned and often reduced over training. Training instability — loss suddenly spikes, gradients explode to NaN, or training plateaus because gradients vanish before reaching early layers. Requires careful hyperparameter tuning and gradient monitoring. LSTM / GRU substantially mitigate this; LayerNorm + residual connections in Transformers make training nearly unconditionally stable.
