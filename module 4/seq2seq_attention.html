Seq2Seq & Attention — Solving the Bottleneck

LSTM solved vanishing gradients, but it introduced a new problem: in encoder-decoder translation, the entire source sentence must be compressed into a single fixed-size vector before decoding begins. For long sentences this bottleneck destroys information. Attention eliminates it entirely — instead of a single summary vector, the decoder learns to dynamically query all encoder states at each step, reading exactly the parts of the source that are relevant right now.

01
The Encoder-Decoder Architecture
How two LSTMs are chained into a sequence-to-sequence system for translation and generation
▾

Many of the most important tasks in sequence modelling involve producing an output sequence whose length is different from the input sequence — and whose structure bears no token-to-token alignment with it. Machine translation ("The cat sat on the mat" → "La gata estaba sentada en la alfombra") is the canonical example, but the same architecture applies to text summarisation (long article → short summary), question answering (question + context → answer), image captioning (image pixels → sentence), and speech recognition (audio frames → text). All of these tasks share one property: the model needs to first read and understand the entire input before it can produce the output.

The encoder-decoder architecture, introduced by Sutskever, Vinyals and Le (2014) and Cho et al. (2014) simultaneously, captures this intuition directly. The encoder is an LSTM (or GRU) that reads the input sequence token by token and accumulates it into its hidden state. No output is produced yet — the encoder's entire job is to build a rich, compressed representation of the input. When the encoder reaches the end of the input, its final hidden state h_T becomes the context vector c — a dense numerical summary of everything the encoder read. The decoder is a second LSTM that then generates the output sequence step by step, initialised with this context vector.

Encoder — reads source tokens x₁…x_T, produces context vector c
$$\mathbf{h}_t^{enc} = \text{LSTM}(\mathbf{x}_t,\, \mathbf{h}_{t-1}^{enc}) \quad t = 1, \ldots, T$$ $$\mathbf{c} = \mathbf{h}_T^{enc} \qquad \text{(context vector = final encoder hidden state)}$$
Decoder — generates target tokens y₁…y_T' conditioned on c
$$\mathbf{s}_0^{dec} = \mathbf{c} \qquad \text{(initialise decoder with context vector)}$$ $$\mathbf{s}_t^{dec} = \text{LSTM}([\mathbf{y}_{t-1},\, \mathbf{c}],\; \mathbf{s}_{t-1}^{dec})$$ $$P(y_t \mid y_{

At each decoding step, the decoder receives two inputs: the previously generated token y_{t−1} (as an embedding), and the context vector c. It produces a new hidden state s_t, which is passed through a linear layer and softmax to produce a probability distribution over the vocabulary. The predicted token is sampled or argmax-selected, then fed back in at the next step. This feedback is called autoregressive generation — the model consumes its own outputs.

During training, a technique called teacher forcing is used: instead of feeding the model's own predicted tokens back in, we feed the ground-truth target tokens. This stabilises training dramatically — early in training, predicted tokens are garbage, and feeding garbage back in causes the decoder to spiral. At inference time, no ground truth is available, so the model uses its own predictions, creating a gap called the exposure bias that is an active area of research.

Encoder-decoder data flow — encoder reads source, decoder generates target token by token
📖 Special Tokens — <SOS>, <EOS>, <PAD>
Three special tokens are essential for sequence-to-sequence training. <SOS> (Start of Sequence) is fed as the first decoder input — it signals "start generating now". <EOS> (End of Sequence) is the token the model learns to generate when it has finished; at inference time, generation stops when EOS is sampled. <PAD> (Padding) fills shorter sequences in a batch to the length of the longest, and positions with PAD are masked out during loss computation with ignore_index. The embedding for PAD is often frozen at zero to prevent it from interfering with the learned token representations.
02
The Fixed-Context Bottleneck — Where Seq2Seq Fails
Why a single vector cannot hold a long sentence — visualised with translation quality vs sentence length
▾

The encoder-decoder architecture is elegant, but it has a fundamental information-theoretic problem. The encoder compresses an input sequence of arbitrary length into a context vector c of fixed size. If the hidden size is 512, then c is a 512-dimensional real-valued vector. This is the only channel through which the entire input is communicated to the decoder — every word, every clause, every grammatical relationship, every long-range dependency must somehow fit inside those 512 numbers.

For short sentences — five to ten words — this is manageable. The LSTM encoder can pack the essential information into c without too much loss. But as sentence length grows, the bottleneck tightens. The encoder must overwrite its hidden state with new information at every step; words seen early in a long sentence are progressively diluted as more words follow. By the time the encoder reaches the end of a 40-word sentence, information about the first few words has been through 35+ rounds of LSTM gating — not completely erased (LSTM is much better than vanilla RNN here), but substantially compressed and mixed with everything else.

The compression problem — a fixed-size vector must hold all information from an arbitrarily long sequence

This degradation was empirically confirmed by Bahdanau et al. (2015), who showed that the BLEU score (a translation quality metric) of a vanilla encoder-decoder model drops sharply as sentence length increases beyond about 20 tokens, while their attention-augmented model maintains quality on long sentences. The fix they proposed — allowing the decoder to look back at all encoder hidden states, not just the last one — is the subject of the next section.

🔴 The Core Problem, Precisely Stated
When generating target token y_t, the decoder needs access to whichever source tokens are most relevant to that particular output. For example, when generating the French word for "cat", the decoder needs to look at the English word "cat" in the source — which might have been processed by the encoder 15 steps ago and has since been substantially overwritten. The fixed context vector c = h_T cannot provide this targeted lookup: it is an undifferentiated mixture of everything the encoder ever read. What we need is a mechanism that lets the decoder say: "for this particular decoding step, I want to focus on these source positions". That is precisely what attention provides.
03
The Attention Idea — Dynamic Context from All Encoder States
Deriving attention from first principles: instead of one fixed c, compute a fresh, weighted context at every decoding step
▾

The bottleneck problem has a straightforward diagnosis: the decoder gets only c = h_T. It cannot look back at intermediate encoder states h_1, h_2, …, h_{T−1} — the rich per-token representations computed as the encoder processed each source word. The fix is almost equally straightforward: don't throw those intermediate states away. Keep all T encoder hidden states, and at each decoding step, let the decoder look at all of them and decide for itself which ones are most useful right now.

More concretely, the attention mechanism replaces the single fixed context vector c with a step-specific context vector c_t that is recomputed fresh at every decoding step t. It is a weighted sum of all encoder hidden states:

Attention context vector — a different weighted sum of encoder states at every decoding step
$$\mathbf{c}_t = \sum_{j=1}^{T} \alpha_{t,j}\, \mathbf{h}_j^{enc}$$

Here α_{t,j} is the attention weight — a scalar between 0 and 1 representing how much the decoder, at step t, should focus on encoder state h_j. The weights satisfy Σ_j α_{t,j} = 1 (they are a probability distribution over source positions), so c_t is literally a probability-weighted average over all encoder hidden states. If α_{t,j} is high for a particular j, the context vector c_t will be dominated by h_j^enc — the decoder is effectively "reading" source position j.

The natural next question is: how are the attention weights α_{t,j} computed? They need to depend on the current decoder state (what the decoder is trying to generate right now) and each encoder state (what the encoder remembers about each source position). The basic recipe is: compute a score e_{t,j} measuring the relevance of encoder state h_j to decoder state s_{t−1}, then normalise all scores with softmax to produce the weights.

From scores to attention weights — three steps
$$e_{t,j} = \text{score}(\mathbf{s}_{t-1},\, \mathbf{h}_j^{enc}) \qquad \text{(1) compute raw relevance scores}$$ $$\alpha_{t,j} = \frac{\exp(e_{t,j})}{\sum_{k=1}^{T} \exp(e_{t,k})} \qquad \text{(2) softmax → probability distribution}$$ $$\mathbf{c}_t = \sum_{j=1}^{T} \alpha_{t,j}\, \mathbf{h}_j^{enc} \qquad \text{(3) weighted sum → context vector}$$
🧠 Analogy — The Translator's Wandering Eye
Imagine a human translator working from English to French. They do not memorise the entire English sentence and then translate from memory (that is the vanilla encoder-decoder). Instead, they keep the English text in front of them and glance at different parts of it as they write each French word. When writing the French word for "cat", they look at the English word "cat". When writing the French past tense verb, they glance at "sat". When constructing the prepositional phrase, they look at "on the mat". Their gaze is content-driven — they look at whichever English words are relevant to the current output word. Attention does exactly this: the "gaze" at each decoding step is the attention distribution α_{t,·}, and it is computed dynamically based on what is being generated.

Why Softmax? Why a Weighted Sum?

The softmax normalisation serves two purposes. First, it converts arbitrary real-valued scores (which can be negative, large, or small in any direction) into a proper probability distribution that sums to 1 and is interpretable as "how much attention to pay". Second, softmax is differentiable everywhere — the attention weights are computed as a smooth function of the scores, and gradients can flow through the entire attention computation during backpropagation. This means the score function's parameters are trained end-to-end alongside the rest of the encoder-decoder: the model learns what to pay attention to purely from the training signal.

The weighted sum rather than, say, argmax selection (just pick the most relevant encoder state) preserves differentiability and allows soft, partial attention. Instead of forcing the model to make a hard binary choice ("I attend to position 3 only"), the weighted sum lets it blend information from multiple source positions — useful when a single target word corresponds to multiple source words (as in idioms, or multiword expressions).

✅ The Aha Moment — Attention is a Differentiable Lookup
Think of the encoder hidden states h_1, …, h_T as entries in a database. The attention mechanism is a soft, differentiable query against this database: the current decoder state s_{t−1} is the query, the encoder states are both the keys (used to compute match scores) and the values (used to form the output). The model learns what a "good match" means entirely from data. This framing — query, key, value — will become central in Transformers, where attention is generalised further. The Seq2Seq attention mechanism here is the conceptual seed of the multi-head self-attention that powers GPT and BERT.

Bahdanau Attention — The Original Additive Mechanism
Bahdanau et al. (2015): a learned alignment network that scores decoder state against each encoder state
▾

The first paper to propose neural attention for machine translation was Bahdanau, Cho and Bengio (2015), "Neural Machine Translation by Jointly Learning to Align and Translate". Their key contribution was both the idea of using all encoder states (not just the last one) and a specific, learnable score function for measuring how well a decoder state and an encoder state align. Their score function is called additive attention or the Bahdanau score:

Bahdanau (additive) score function — a small feedforward network
$$e_{t,j} = \mathbf{v}_a^\top \tanh\!\left(\mathbf{W}_a\,\mathbf{s}_{t-1} + \mathbf{U}_a\,\mathbf{h}_j^{enc}\right)$$

There are three learned parameter matrices here: W_a projects the decoder state s_{t−1}, U_a projects the encoder state h_j, the two projections are added together and passed through tanh, and then a learned vector v_a takes the dot product to produce a scalar score. This is essentially a two-layer network (linear → tanh → linear) that takes two vectors as input and produces a scalar measuring their compatibility. The tanh nonlinearity allows the network to model interactions between the decoder and encoder representations — if both have a high value in some dimension, or if they have complementary values, the network can detect this.

Notice that U_a h_j^enc does not depend on the decoder step t. This means we can pre-compute all the encoder projections U_a h_j^enc before decoding starts, and only compute W_a s_{t−1} freshly at each decoding step. This is an important efficiency optimisation: computing T encoder projections once rather than T × T' times (once per decoding step per source position).

Bahdanau et al. also used a bidirectional LSTM encoder — running one LSTM forward over the source sequence and another backward, then concatenating their hidden states at each position. This gives h_j a representation that incorporates context from both directions: what came before position j and what comes after. A word's meaning often depends on its surrounding context, and bidirectionality lets the encoder capture this before the attention mechanism tries to retrieve it.

Bahdanau attention — full computation graph for one decoding step t

The complete Bahdanau attention-augmented decoder receives c_t at every step, concatenated with the embedded previous token, to produce the new decoder hidden state:

Bahdanau decoder update — context vector recomputed at every step t
$$\mathbf{s}_t = \text{LSTM}\!\left([\mathbf{y}_{t-1},\; \mathbf{c}_t],\; \mathbf{s}_{t-1}\right)$$ $$P(y_t) = \text{softmax}\!\left(\mathbf{W}\,[\mathbf{s}_t,\, \mathbf{c}_t,\, \mathbf{y}_{t-1}]\right)$$

The output projection often concatenates s_t, c_t, and y_{t−1} all together — giving the prediction access to the current decoder state, the source context, and the previous output all simultaneously. This "deep output layer" structure consistently helps translation quality.

05
Attention Score Functions — How to Measure Relevance
Additive (Bahdanau), dot-product, scaled dot-product, and general — trade-offs in expressiveness vs speed
▾

The score function e_{t,j} = score(s_{t−1}, h_j) is the heart of the attention mechanism — it determines what "relevant" means. Different choices of score function correspond to different assumptions about what kind of similarity between decoder and encoder states should drive attention. After Bahdanau's additive attention, Luong et al. (2015) proposed several alternatives that are simpler and faster. Understanding why each works illuminates the deeper structure of attention itself.

Name Formula Intuition & Trade-off
Additive
(Bahdanau)		 v_a · tanh(W_a s + U_a h) A small neural network measures compatibility. Expensive: two separate projections + nonlinearity. Most expressive — can capture complex, non-linear relationships between s and h. Requires 3 learned matrices.
Dot-product
(Luong)		 s · h Pure inner product — measures cosine-like alignment if both vectors are normalised. Zero parameters! Fastest possible. Requires s and h to be the same size. Works surprisingly well when encoder and decoder use the same hidden dimension.
Scaled dot-product
(Transformer)		 s · h / √d Dot product divided by √d (where d = hidden dimension). Prevents scores from growing large in magnitude as d increases — otherwise softmax becomes extremely peaked, gradients saturate and learning stalls. This is the attention used in Transformers. Zero parameters, same speed as dot-product.
General
(Luong)		 s · W_a · h A bilinear form: one learned matrix W_a mediates the interaction. More expressive than dot-product (allows different encoder and decoder dimensions), less expensive than additive (no nonlinearity). A good middle ground — 1 learned matrix, no nonlinearity.
Concat
(Luong)		 v · tanh(W_a [s;h]) Very similar to Bahdanau's additive — concatenate instead of adding projected vectors. The key difference: Bahdanau adds W_a s + U_a h (two matrices, can handle different dimensions); concat uses a single W_a [s;h] (one matrix on a 2d-dimensional input). Practically equivalent performance.

Why Scaling by √d Matters

The motivation for the scaling factor in scaled dot-product attention is worth understanding precisely. Consider two d-dimensional vectors s and h whose components are independently drawn from N(0,1). Their dot product s · h = Σ s_i h_i is a sum of d independent random variables, each with mean 0 and variance 1. By the central limit theorem, this sum has variance d and standard deviation √d. As d increases — say from 64 to 512 — the dot product's magnitude grows proportionally with √d, pushing softmax inputs into regions where the function saturates near 0 or 1. In those regions, gradients vanish. Dividing by √d normalises the variance back to 1 regardless of d, keeping the softmax in a healthy, gradient-rich operating regime. This small detail — dividing by √d — makes the difference between stable training and collapsing gradients in large models.

Scaled dot-product — the correct attention in most modern architectures
$$e_{t,j} = \frac{\mathbf{s}_{t-1} \cdot \mathbf{h}_j}{\sqrt{d}} \qquad \text{where } d = \text{dimension of hidden states}$$ $$\alpha_{t,j} = \text{softmax}_j\!\left(\frac{\mathbf{s}_{t-1} \cdot \mathbf{h}_j}{\sqrt{d}}\right)$$
✅ Which Score Function to Use in Practice
For standard sequence-to-sequence with an LSTM encoder-decoder, additive (Bahdanau) attention is the most common choice — it is robust, allows different encoder/decoder hidden sizes, and the extra computation is negligible for typical sequence lengths. If you are building something Transformer-adjacent or care about speed, use scaled dot-product. The general (bilinear) attention is a clean middle ground if you want one learned parameter with no nonlinearity. In practice, the choice rarely changes results dramatically — getting the architecture right and training stably matters far more than which score function you pick.

Luong Attention — Global vs Local, and a Cleaner Decoder Design
Luong et al. (2015): simplifying the architecture by computing attention after the LSTM step, not before
▾

Luong, Pham and Manning (2015) introduced a set of attention variants that are architecturally cleaner in one important respect: whereas Bahdanau attention computes the context vector c_t before the decoder LSTM step (using s_{t−1} to query encoder states, then feeding c_t into the LSTM), Luong attention computes attention after the LSTM step. The decoder first runs its LSTM to produce a new hidden state h̃_t using only the previous hidden state and the current input, and then uses h̃_t to query the encoder states and compute c_t. Finally, c_t and h̃_t are combined to produce the final hidden state used for the output projection.

Luong attention — attention computed after the LSTM step (not before)
$$\tilde{\mathbf{h}}_t = \text{LSTM}(\mathbf{y}_{t-1},\, \mathbf{s}_{t-1}) \qquad \text{(1) run decoder LSTM first}$$ $$e_{t,j} = \text{score}(\tilde{\mathbf{h}}_t,\, \mathbf{h}_j^{enc}) \qquad \text{(2) compute scores using new state}$$ $$\mathbf{c}_t = \sum_j \alpha_{t,j}\, \mathbf{h}_j^{enc} \qquad \text{(3) weighted context}$$ $$\mathbf{\hat{h}}_t = \tanh(\mathbf{W}_c\,[\mathbf{c}_t;\, \tilde{\mathbf{h}}_t]) \qquad \text{(4) combine context and LSTM state}$$ $$P(y_t) = \text{softmax}(\mathbf{W}_o\, \mathbf{\hat{h}}_t) \qquad \text{(5) predict output}$$

This ordering — LSTM first, then attention — makes the computation slightly cleaner because the LSTM can run independently, and the attention acts as a post-processing enrichment step rather than being woven into the LSTM's inputs. In practice the performance difference between Bahdanau and Luong is small; the choice often comes down to implementation preference.

Global vs Local Attention

Luong et al. also introduced a useful distinction between global attention and local attention. Global attention is what we have been discussing: the decoder attends over all T source positions at every step, computing a weighted sum over all encoder states. This has O(T) cost per decoding step — for very long documents (T in the thousands), this becomes expensive. Local attention addresses this by attending over only a window of p source positions centred around a predicted alignment position p_t. This reduces computation to O(p) per step, which is crucial for document-level translation or speech recognition where source sequences can be very long.

📖 Bahdanau vs Luong — Architectural Summary
Bahdanau: uses s_{t−1} (state before step t) to compute attention, then feeds c_t into the LSTM. The attention is "predictive" — it decides what context is needed before the recurrent step runs. Luong: runs the LSTM step first (using only the input and previous state), then uses the resulting h̃_t to compute attention, then combines h̃_t and c_t for the output. Attention is "retrospective" — it enriches the hidden state after it has been computed. Both work well in practice. For a new project, Luong's formulation is slightly simpler to implement cleanly.

Visualising Attention Weights — What the Model Learns to Look At
Interpreting the attention matrix, the near-diagonal pattern in translation, and what non-monotonic attention reveals
▾

One of the most revealing properties of attention-based models is that the attention weights α_{t,j} can be visualised as a matrix — rows correspond to decoder steps (target positions), columns correspond to encoder steps (source positions), and cell intensity represents how much the decoder attended to each source position when generating each target token. These visualisations were first shown by Bahdanau et al. and have since been one of the most powerful tools for understanding and debugging sequence-to-sequence models.

Attention matrix — English→French translation (simplified). Rows=target, columns=source, darker=higher attention

Several patterns emerge from reading this attention matrix. The dominant pattern in translation is a near-diagonal band — most target tokens attend primarily to the source token at roughly the corresponding position. This makes linguistic sense: most words in a translation correspond to approximately the same position in the source. The model has learned this near-monotonic alignment without any explicit supervision about word alignments — it emerged purely from the translation objective.

The more interesting cells are the off-diagonal ones. In the example above, when generating the French word "noir" (black), the model attends strongly to the English word "black" — but "black" comes before "cat" in English, while "noir" comes after "chat" (cat) in French. The attention matrix has learned the English-French adjective-noun word order reordering automatically. Similarly, verbs, prepositions, and articles that have different positions in source and target languages create characteristic off-diagonal patterns that have been studied by linguists to verify that the model is learning genuine linguistic structure.

🧠 Attention as a Soft Alignment
In statistical machine translation (the pre-neural approach), word alignment was a central concept: a learned function that mapped each source word to the target word(s) that translated it. Computing these alignments required complex, hand-designed models (IBM Model 1 through 5, HMM alignment models). Neural attention learns an equivalent representation — the α_{t,j} matrix is essentially a soft alignment — purely as a byproduct of training to maximise translation likelihood. No alignment data is ever provided. The model discovers what aligns with what through gradient descent alone. This was considered remarkable in 2015 and remains one of the cleanest demonstrations of representation learning working as hoped.

PyTorch Implementation — Bahdanau Attention Seq2Seq
Encoder, attention module, decoder, and training loop — built piece by piece with every design choice explained
▾

We will build a complete attention-based encoder-decoder in PyTorch in four pieces. Reading each piece in order mirrors the data flow at inference time: tokens flow into the encoder, all encoder states are saved, the attention module computes context vectors for each decoder step, and the decoder generates outputs autoregressively. Every design choice below is annotated with the why, not just the what.

Part 1 — Bidirectional Encoder

Python · Encoder — BiLSTM, keeps ALL hidden states for attention
import torch
import torch.nn as nn
import torch.nn.functional as F

class Encoder(nn.Module):
    """BiLSTM encoder that returns ALL hidden states (not just the final one).
    
    Key difference from a vanilla encoder-decoder:
    - old: return only h_T (the final hidden state → context vector)
    - now: return ALL h_1, h_2, ..., h_T for the attention mechanism to query.
    
    We also return h_T to initialise the decoder hidden state.
    BiLSTM gives each position both left-context and right-context.
    """
    def __init__(self, vocab_size, embed_dim, hidden_size,
                 num_layers=1, dropout=0.3):
        super().__init__()
        self.hidden_size = hidden_size
        self.embed = nn.Embedding(vocab_size, embed_dim, padding_idx=0)
        self.lstm  = nn.LSTM(
            embed_dim, hidden_size, num_layers,
            batch_first=True, bidirectional=True,
            dropout=dropout if num_layers > 1 else 0.0
        )
        # Project bidirectional hidden (hidden*2) back to hidden
        # for the decoder's initial state (decoder expects hidden, not hidden*2)
        self.fc_h = nn.Linear(hidden_size * 2, hidden_size)
        self.fc_c = nn.Linear(hidden_size * 2, hidden_size)

    def forward(self, src):
        """
        src: (batch, src_len) — source token IDs
        Returns:
          enc_out:  (batch, src_len, hidden*2) — ALL hidden states
          h_init:   (1, batch, hidden) — projected initial state for decoder
          c_init:   (1, batch, hidden)
        """
        x = self.embed(src)                   # (batch, src_len, embed)
        enc_out, (h_n, c_n) = self.lstm(x)   # enc_out: (batch, src_len, hidden*2)

        # h_n shape: (num_layers*2, batch, hidden) for BiLSTM
        # Take the last layer's forward (h_n[-2]) and backward (h_n[-1]),
        # concatenate them, and project to decoder's hidden size.
        h_init = torch.tanh(self.fc_h(
            torch.cat([h_n[-2], h_n[-1]], dim=1)
        )).unsqueeze(0)                      # (1, batch, hidden)

        c_init = torch.tanh(self.fc_c(
            torch.cat([c_n[-2], c_n[-1]], dim=1)
        )).unsqueeze(0)                      # (1, batch, hidden)

        return enc_out, h_init, c_init

Part 2 — Bahdanau Attention Module

Python · Bahdanau (additive) attention — score, softmax, weighted sum
class BahdanauAttention(nn.Module):
    """Computes a step-specific context vector from all encoder states.
    
    At every decoder step t, this module:
      1. Scores each encoder hidden state against the current decoder state
      2. Normalises scores with softmax → attention weights α (sum to 1)
      3. Returns weighted sum of encoder states → context vector c_t
    
    The score function: e_{t,j} = v_a · tanh(W_a * dec_hidden + U_a * enc_out)
    W_a and U_a project to the same attention_dim so they can be summed.
    """
    def __init__(self, enc_hidden_size, dec_hidden_size, attn_dim=256):
        super().__init__()
        # W_a: projects decoder state from dec_hidden → attn_dim
        self.W_a = nn.Linear(dec_hidden_size, attn_dim, bias=False)
        # U_a: projects encoder states from enc_hidden_size → attn_dim
        # enc_hidden_size is hidden*2 (bidirectional)
        self.U_a = nn.Linear(enc_hidden_size, attn_dim, bias=False)
        # v_a: scalar projection from attn_dim → 1 score per position
        self.v_a = nn.Linear(attn_dim, 1, bias=False)

    def forward(self, dec_hidden, enc_out):
        """
        dec_hidden: (batch, dec_hidden_size) — current decoder state s_{t-1}
        enc_out:    (batch, src_len, enc_hidden_size) — all encoder states
        Returns:
          context:  (batch, enc_hidden_size) — weighted sum c_t
          attn_w:   (batch, src_len) — attention weights (for visualisation)
        """
        # Project decoder hidden state: (batch, attn_dim)
        # Unsqueeze to broadcast over src_len dimension
        dec_proj = self.W_a(dec_hidden).unsqueeze(1)  # (batch, 1, attn_dim)

        # Project all encoder states: (batch, src_len, attn_dim)
        # This can be pre-computed once before the decoding loop!
        enc_proj = self.U_a(enc_out)               # (batch, src_len, attn_dim)

        # Add and apply tanh — broadcasting handles the dimensions
        # Result: (batch, src_len, attn_dim)
        combined = torch.tanh(dec_proj + enc_proj)

        # Scalar score per encoder position: (batch, src_len, 1) → squeeze
        scores = self.v_a(combined).squeeze(-1)     # (batch, src_len)

        # Normalise to probability distribution over source positions
        attn_w = F.softmax(scores, dim=-1)          # (batch, src_len)

        # Weighted sum of encoder states → context vector
        # bmm: batch matrix multiply
        # attn_w: (batch, 1, src_len) × enc_out: (batch, src_len, enc_h)
        # result: (batch, 1, enc_h) → squeeze to (batch, enc_h)
        context = torch.bmm(
            attn_w.unsqueeze(1), enc_out
        ).squeeze(1)                               # (batch, enc_hidden_size)

        return context, attn_w

Part 3 — Attention Decoder

Python · Decoder — LSTM + attention, step-by-step with teacher forcing
class AttentionDecoder(nn.Module):
    """One-step decoder: takes previous token + previous hidden state + encoder
    outputs, attends to encoder, and predicts the next token.
    
    Note: this is a single-step module. The sequence loop (with teacher
    forcing or autoregression) lives in the Seq2Seq wrapper below.
    """
    def __init__(self, vocab_size, embed_dim, dec_hidden_size,
                 enc_hidden_size, attn_dim=256, dropout=0.3):
        super().__init__()
        self.embed   = nn.Embedding(vocab_size, embed_dim, padding_idx=0)
        self.attn    = BahdanauAttention(enc_hidden_size, dec_hidden_size, attn_dim)
        # LSTM input: concatenated [embed(y_{t-1}), context_vector]
        self.lstm    = nn.LSTM(
            embed_dim + enc_hidden_size, dec_hidden_size,
            batch_first=True
        )
        self.drop    = nn.Dropout(dropout)
        # Output projection: [s_t, c_t, embed(y_{t-1})] → vocab (deep output)
        self.fc_out  = nn.Linear(
            dec_hidden_size + enc_hidden_size + embed_dim, vocab_size
        )

    def forward(self, prev_token, h_prev, c_prev, enc_out):
        """
        prev_token: (batch,) — previous output token (or SOS for first step)
        h_prev, c_prev: (1, batch, dec_hidden) — decoder LSTM states
        enc_out: (batch, src_len, enc_hidden*2) — all encoder hidden states
        """
        emb = self.embed(prev_token.unsqueeze(1))     # (batch, 1, embed)

        # Compute attention context using PREVIOUS decoder hidden state
        # h_prev: (1, batch, hidden) → squeeze to (batch, hidden) for attention
        context, attn_w = self.attn(h_prev.squeeze(0), enc_out)

        # Feed [embedding, context] into LSTM
        # context unsqueezed to (batch, 1, enc_h) to concat with emb (batch, 1, embed)
        lstm_in = torch.cat([emb, context.unsqueeze(1)], dim=-1)
        out, (h_new, c_new) = self.lstm(lstm_in, (h_prev, c_prev))

        out = out.squeeze(1)  # (batch, dec_hidden)

        # Deep output: concatenate [s_t, c_t, embed(y_{t-1})]
        # This gives the prediction direct access to all three information sources
        logits = self.fc_out(self.drop(
            torch.cat([out, context, emb.squeeze(1)], dim=-1)
        ))                                              # (batch, vocab_size)

        return logits, h_new, c_new, attn_w

Part 4 — Seq2Seq Wrapper with Teacher Forcing

Python · Seq2Seq wrapper — encodes source, decodes with scheduled teacher forcing
class Seq2SeqAttention(nn.Module):
    def __init__(self, encoder, decoder, device):
        super().__init__()
        self.encoder = encoder
        self.decoder = decoder
        self.device  = device

    def forward(self, src, trg, teacher_forcing_ratio=0.5):
        """
        src:  (batch, src_len)
        trg:  (batch, trg_len) — includes SOS at trg[:,0]
        teacher_forcing_ratio: probability of using ground-truth instead of
                               predicted token at each step. Start high (0.9)
                               and decay over training toward 0.3-0.5.
        """
        batch_size = src.shape[0]
        trg_len    = trg.shape[1]
        vocab_size = self.decoder.fc_out.out_features

        # ── Encode the full source sentence ────────────────────────────
        enc_out, h, c = self.encoder.forward(src)
        # enc_out: (batch, src_len, hidden*2) — kept for attention at every step

        # ── Decode token-by-token ──────────────────────────────────────
        all_logits  = torch.zeros(batch_size, trg_len - 1, vocab_size).to(self.device)
        all_attn    = []
        token       = trg[:, 0]  # first input = SOS token

        for t in range(1, trg_len):
            logits, h, c, attn_w = self.decoder.forward(token, h, c, enc_out)
            all_logits[:, t - 1] = logits
            all_attn.append(attn_w.detach())

            # Teacher forcing: use ground-truth token with probability `ratio`
            # At inference, always use model's own prediction (ratio = 0.0)
            use_teacher = torch.rand(1).item() < teacher_forcing_ratio
            token = trg[:, t] if use_teacher else logits.argmax(dim=-1)

        # Stack attention matrices: (batch, trg_len-1, src_len)
        attn_matrix = torch.stack(all_attn, dim=1)
        return all_logits, attn_matrix


# ── Training loop ────────────────────────────────────────────────
SRC_VOCAB, TRG_VOCAB = 10000, 10000
device  = torch.device('cuda' if torch.cuda.is_available() else 'cpu')

enc = Encoder(SRC_VOCAB, embed_dim=256, hidden_size=512).to(device)
dec = AttentionDecoder(TRG_VOCAB, embed_dim=256, dec_hidden_size=512,
                       enc_hidden_size=1024).to(device)  # enc_h*2 = 1024
model    = Seq2SeqAttention(enc, dec, device).to(device)
optimizer= torch.optim.Adam(model.parameters(), lr=3e-4)
criterion= nn.CrossEntropyLoss(ignore_index=0)  # ignore PAD tokens

for epoch in range(30):
    for src, trg in train_loader:
        src, trg = src.to(device), trg.to(device)
        optimizer.zero_grad()

        # logits: (batch, trg_len-1, vocab)
        # reshape for cross-entropy: (batch*(trg_len-1), vocab)
        logits, _ = model.forward(src, trg, teacher_forcing_ratio=0.5)
        loss = criterion(
            logits.view(-1, TRG_VOCAB),
            trg[:, 1:].contiguous().view(-1)  # skip SOS in targets
        )
        loss.backward()
        nn.utils.clip_grad_norm_(model.parameters(), 5.0)
        optimizer.step()
⚠️ Greedy Decoding vs Beam Search
The inference loop above uses greedy decoding — always pick the highest-probability token at each step. This is fast but suboptimal: the globally best translation might require choosing a lower-probability word early that enables much better words later. Beam search maintains a set of k candidate sequences (the "beam"), expanding all of them at each step and keeping only the top k by cumulative log-probability. k=4 or k=8 is standard for translation. Implementing beam search is tedious but important for production quality. For a quick prototype, greedy decoding is fine; for a publication-quality result, beam search is expected.

Applications — Translation, Summarisation, Image Captioning
How the same attention architecture adapts to different input modalities and task structures
▾

The attention-augmented encoder-decoder was one of the most generalisable architectures in deep learning history. Once the core idea was established — keep all encoder states and let the decoder attend to them — it spread to virtually every sequence generation task. The encoder does not even need to produce a text sequence: it can encode an image (using a CNN), an audio signal (using a spectrogram), or any other structured input, and as long as you can represent the encoding as a sequence of vectors, the attention decoder works unchanged.

🌐
Machine Translation
The original application. BiLSTM encoder over source tokens; LSTM decoder with Bahdanau attention generates target tokens. Attention handles word-reordering between languages naturally. Google Translate used this architecture in production before Transformers. BLEU scores on WMT benchmarks improved dramatically over vanilla seq2seq.
📰
Text Summarisation
Encoder reads a long document, decoder generates a shorter summary. Attention is critical: the decoder must selectively attend to the most important sentences in the source. A "coverage mechanism" was added — a running sum of past attention weights that penalises attending to the same positions repeatedly, encouraging broad coverage of the source.
🖼️
Image Captioning
CNN encoder extracts a grid of feature vectors (one per spatial region). LSTM decoder with attention generates a caption word by word. At each step, the decoder attends to different regions of the image — "a" and "the" attend broadly, while "cat" attends to the cat region and "mat" attends to the floor. Xu et al. (2015) showed convincing spatial attention maps.
🎤
Speech Recognition
Encoder processes audio features (log-mel spectrograms); decoder generates text tokens. Attention learns the monotonic alignment between audio frames and phonemes/characters. The model does not need forced alignment labels — it learns the correspondence from (audio, transcript) pairs alone. Listen, Attend and Spell (LAS) was the seminal architecture.

Image Captioning — Attention Over Spatial Regions

Image captioning with attention is a particularly clean demonstration of what the mechanism learns, because you can visualise exactly where the model is "looking" when it generates each word. In the Show, Attend and Tell architecture (Xu et al., 2015), a CNN encodes an image into a grid of feature vectors — say 14×14=196 spatial positions, each a 512-dimensional vector. The LSTM decoder then attends over these 196 spatial vectors at each time step, producing a context vector that is dominated by the most visually relevant region for the current word.

Python · Image captioning encoder — CNN spatial features as "encoder states"
import torchvision.models as models

class ImageEncoder(nn.Module):
    """Extract spatial feature grid from an image using a pretrained CNN.
    
    Instead of taking the final pooled vector (which would be the fixed
    bottleneck again), we take the last convolutional layer's feature map:
    shape (batch, C, H, W) → reshape to (batch, H*W, C)
    This gives us H*W spatial positions, each with C features — exactly
    the "sequence of vectors" that attention expects.
    """
    def __init__(self, embed_dim=512):
        super().__init__()
        resnet = models.resnet50(pretrained=True)
        # Remove the average pooling and fc layers — keep spatial resolution
        self.features = nn.Sequential(*list(resnet.children())[:-2])
        self.proj     = nn.Linear(2048, embed_dim)

        # Freeze early layers (they detect edges/textures, universal across tasks)
        for param in list(self.features.parameters())[:-20]:
            param.requires_grad = False

    def forward(self, images):
        """
        images: (batch, 3, 224, 224)
        Returns: (batch, num_pixels, embed_dim) — the "encoder output"
                  that the attention decoder will query at each caption step
        """
        feats = self.features(images)          # (batch, 2048, 7, 7)
        batch, C, H, W = feats.shape
        feats = feats.view(batch, C, H * W)    # (batch, 2048, 49)
        feats = feats.permute(0, 2, 1)          # (batch, 49, 2048)
        return self.proj(feats)                 # (batch, 49, embed_dim)
        # ↑ This is now shaped exactly like text encoder output (batch, seq, dim)
        # The same BahdanauAttention module works unchanged!
✅ The Generality of the Attention Interface
The fact that the same BahdanauAttention module works unchanged for text translation and image captioning — simply because both produce a tensor of shape (batch, seq_len, hidden_dim) — reveals something deep about the architecture. Attention does not care about what the encoder states represent. It only cares about their shape. You could encode video frames, protein sequences, molecules, or music spectrograms with the appropriate encoder, and the same attention decoder would work out of the box. This modality-agnostic quality is a big part of why attention became the foundation for modern multimodal models.

What Attention Still Can't Solve — The Door to Transformers
Recurrence as the remaining bottleneck: O(T) sequential computation, path length between distant tokens, and what self-attention changes
▾

Attention with LSTM encoder-decoders was a transformative development, but it only half-solved the problem of sequence modelling. It solved the information bottleneck — the decoder can now read any encoder state, not just h_T. But it did not solve the computational structure of the encoder itself. The encoder is still an LSTM: it processes the source sequence one token at a time, and step t cannot begin until step t−1 is finished. This sequential dependency is not a practical annoyance — it is a fundamental architectural constraint that limits both speed and the quality of representations the encoder can build.

Problem 1 — Sequential Computation Cannot Be Parallelised

Modern GPU hardware achieves peak performance through massive parallelism — thousands of cores executing the same operation on different data simultaneously. A matrix multiplication over a batch of sequences can be parallelised perfectly, using all available cores. But an LSTM step cannot: the computation of h_t requires h_{t−1}, which requires h_{t−2}, and so on. The LSTM must execute its T steps in sequence, using only one step's worth of parallelism at a time. For a sequence of length T=512, this means 512 sequential LSTM steps, each one waiting for the previous to finish. On a GPU with 10,000 cores, only a fraction of the hardware is ever utilised.

This is not merely a speed concern. Training time directly limits how much data and how many experiments researchers can run. If training a translation model takes 2 weeks instead of 2 days, the number of architectures you can explore, the amount of data you can use, and the number of hyperparameter configurations you can try all shrink by 7×. The Transformer's fully parallel computation — the key architectural change introduced in "Attention is All You Need" (Vaswani et al., 2017) — was critical to the explosion of model scale and capability that followed.

Sequential vs parallel — LSTM must process tokens one by one; Transformer processes all positions simultaneously

Problem 2 — Long Path Length Between Distant Tokens

Even with attention, there is a second structural problem. Consider two tokens at positions 1 and 500 in an LSTM-encoded sequence. To capture any interaction between them, information must flow through the LSTM's hidden state over 499 steps — each step applying the forget, input, and output gates and potentially overwriting or diluting the signal. Even though LSTM is much better than vanilla RNN at this, 499 recurrent steps is a long path for any signal to survive unchanged. In a Transformer, any two positions can directly attend to each other regardless of how far apart they are — the path length between any two positions is exactly 1. This fundamentally changes what kinds of long-range patterns the model can learn.

Problem 3 — The Encoder Cannot Look at Itself

In the Seq2Seq attention model, attention goes from the decoder to the encoder — it is cross-attention. The encoder itself is purely sequential: when computing h_j, the LSTM has only access to x_1, …, x_j (in a unidirectional encoder) or x_1, …, x_T (in a bidirectional encoder, but even then only through the sequential recurrent path). The encoder cannot compute a direct, context-sensitive representation of word j that is enriched by querying all other positions j simultaneously. Self-attention — where each position queries all other positions in the same sequence — allows the encoder to build much richer contextualised representations. When encoding the word "bank", a self-attention encoder can immediately look at both "river bank" context words and weight them appropriately, without relying on the sequential hidden state to have accumulated and preserved that context.

🚀 "Attention Is All You Need" — What Changes in a Transformer
The Transformer (Vaswani et al., 2017) makes one radical architectural decision: remove recurrence entirely. There are no LSTM steps, no hidden states, no sequential dependencies. Instead, the entire encoding is done by self-attention layers — each position queries all other positions simultaneously, and the output at each position is a weighted sum of all input representations (including its own). Because there is no recurrence, all T positions can be processed in parallel. Positional encodings are added to preserve word-order information (which recurrence encoded implicitly). The result is a model that trains much faster, learns longer-range dependencies more effectively, and scales to billions of parameters.
