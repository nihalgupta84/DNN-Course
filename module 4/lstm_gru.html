LSTM & GRU — Solving the Memory Problem

Vanilla RNNs have a memory of roughly 5–10 steps because vanishing gradients erase the signal from the past. LSTM's answer is architectural — a separate cell state that flows through time with only additive updates, keeping the gradient highway open over hundreds of steps. This page builds both LSTM and GRU from first principles, gate by gate.

01
From Vanilla RNN to LSTM — The Design Problem
Why adding gates is the right architectural answer to vanishing gradients
▾

Page 1 ended with a precise diagnosis: vanilla RNNs cannot learn long-range dependencies because the gradient signal decays exponentially as it flows backward through time, multiplying by the recurrent weight matrix W_hh at every step. By the time the gradient reaches an event from 30 steps ago, it has been multiplied by W_hh thirty times — and if the spectral radius of W_hh is less than 1 (which it must be for the network to be stable), this product collapses toward zero.

The naive solution would be to initialise W_hh so that gradients neither vanish nor explode — but this is unstable. A matrix with eigenvalues exactly equal to 1 would preserve gradient magnitude, but during training the eigenvalues drift as weights change, and any small perturbation causes either explosion or vanishing. You cannot keep a learned matrix exactly at the identity.

Hochreiter and Schmidhuber's insight in 1997 was that the problem is not how the recurrent weights are initialised — it is the structure of the update itself. In a vanilla RNN, every step overwrites the hidden state completely:

Vanilla RNN — multiplicative overwrite at every step
$$\mathbf{h}_t = \tanh(\mathbf{W}_{hh}\,\mathbf{h}_{t-1} + \mathbf{W}_{xh}\,\mathbf{x}_t + \mathbf{b})$$

This means information must be re-encoded at every time step. Old information has no direct path to survive — it must somehow survive repeated matrix multiplications, each of which degrades it. The LSTM solution is to create a second, separate memory pathway — the cell state — that flows through the network with only additive updates. Addition preserves gradients; multiplication destroys them. This is the entire conceptual core of LSTM, and every gate exists to control what flows into and out of this additive pathway.

🧠 Analogy — RAM vs a Telephone Game
A vanilla RNN is like the telephone game: each player whispers what they heard to the next player, and after 30 repetitions the original message is unrecognisable. Every transmission is noisy and lossy. An LSTM is like having a shared whiteboard in the room — important information can be written down once and read directly by anyone in the chain without passing through multiple noisy transformations. The cell state is that whiteboard. Gates are the hands that decide what gets written, erased, or read.

The Two-Track Architecture

The key architectural difference between vanilla RNN and LSTM is that LSTM maintains two separate state vectors at every time step: the cell state c_t and the hidden state h_t. These two vectors play completely different roles and flow through the network in fundamentally different ways.

Two tracks: cell state (gradient highway) vs hidden state (working memory)

Notice that the cell state update is fundamentally additive: c_t = f_t ⊙ c_{t−1} + i_t ⊙ g_t. The second term is an addition, not a matrix multiplication. This means the gradient of the loss with respect to c_{t−k} can flow back through the addition operation without being multiplied by any weight matrix — it arrives at c_{t−k} essentially intact. This is sometimes called the constant error carousel, Hochreiter's original term for the mechanism that prevents vanishing gradients.

02
The Cell State — Long-Term Memory Highway
What the cell state is, why it solves the vanishing gradient problem, and how information flows along it
▾

The cell state c_t is a vector of the same dimensionality as the hidden state, but it plays an entirely different role. Think of h_t as the network's working memory — a compressed, processed summary that is used directly for making predictions. Think of c_t as the network's long-term memory — a carefully maintained record of important information that has been accumulated over many time steps, passed forward largely intact, and only selectively modified at each step.

The cell state update equation is the most important equation in the LSTM:

Cell state update — additive, not multiplicative
$$\mathbf{c}_t = \underbrace{\mathbf{f}_t \odot \mathbf{c}_{t-1}}_{\text{what to keep from past}} + \underbrace{\mathbf{i}_t \odot \tilde{\mathbf{c}}_t}_{\text{what new info to add}}$$

The symbol ⊙ denotes element-wise multiplication (the Hadamard product). The two terms correspond to two decisions the network makes at every step: how much of the old cell state to keep (controlled by the forget gate f_t), and how much new information to write in (controlled by the input gate i_t and candidate values c̃_t). The crucial insight is that the second term is an addition — and addition lets gradients pass through unchanged.

Why Addition Preserves Gradients

Consider how backpropagation flows through the cell state update. When we compute ∂L/∂c_{t−1}, the chain rule gives:

Gradient flows backward through the cell state — only element-wise multiplication by f_t
$$\frac{\partial \mathcal{L}}{\partial \mathbf{c}_{t-1}} = \frac{\partial \mathcal{L}}{\partial \mathbf{c}_t} \cdot \frac{\partial \mathbf{c}_t}{\partial \mathbf{c}_{t-1}} = \frac{\partial \mathcal{L}}{\partial \mathbf{c}_t} \odot \mathbf{f}_t$$

Compare this to the vanilla RNN gradient, which multiplied by the full matrix W_hh at every step. Here, the gradient at c_t is multiplied only by the forget gate values f_t — a vector of numbers between 0 and 1, applied element-wise. Crucially, there is no weight matrix multiplication in this path. The gradient can flow hundreds of steps back through the cell state, decaying only by the forget gate values rather than by repeated matrix products. If the forget gate learns to output values close to 1 for an important memory dimension, that dimension's gradient flows back with almost no decay at all — this is the constant error carousel.

✅ The Gradient Highway — Why LSTM Can Remember 1000 Steps
For a dimension where f_t ≈ 1.0 at every step, the gradient ∂L/∂c_{t−1000} ≈ ∂L/∂c_t × (1.0)^1000 = ∂L/∂c_t. The gradient arrives at c_{t−1000} essentially unchanged! This is fundamentally impossible in a vanilla RNN, where the gradient would be multiplied by W_hh^1000, collapsing to zero (or exploding to infinity). LSTM has been shown empirically to retain meaningful gradient signal over sequences of 1,000+ time steps — something that is simply out of reach for vanilla RNNs.

What Does Each Dimension of the Cell State Encode?

Researchers have visualised individual cell state dimensions in trained LSTMs and found remarkably interpretable structure. In a language model trained on text, individual cell state dimensions have been found to track things like: "am I currently inside a quotation?", "how many clauses deep am I in the current sentence?", "has the subject of the current clause been mentioned?", "is the current discourse sentiment positive or negative?". These are not hand-coded — they emerge from learning to predict the next word. The cell state learns to maintain whatever long-range information is statistically useful for predicting the future of the sequence.

📖 Three Gates — Three Decisions
The cell state by itself is just a passive storage mechanism. Three learned gates control what happens to it at every time step. Each gate is a vector of values between 0 and 1, produced by a sigmoid-activated linear layer. A gate value of 0 means "block this completely"; a value of 1 means "let this through unchanged"; intermediate values allow fine-grained partial selection. The three gates are the forget gate (what to erase from c_{t−1}), the input gate (what new information to write into c_t), and the output gate (what to expose from c_t as the working memory h_t). We will build each one in its own section.
03
The Forget Gate — Deciding What to Erase
Gate 1 of 3: how the network learns to selectively clear irrelevant long-term memory
▾

The forget gate is the first decision the LSTM makes at each time step, and it is conceptually the simplest: given what I just saw (x_t) and what my recent working memory says (h_{t−1}), how much of my long-term memory should I discard? It produces a vector of values between 0 and 1 — one per cell state dimension — where 0 means "erase this completely" and 1 means "keep this unchanged".

Forget gate equation
$$\mathbf{f}_t = \sigma\!\left(\mathbf{W}_f\,[\mathbf{h}_{t-1},\, \mathbf{x}_t] + \mathbf{b}_f\right)$$

The notation [h_{t−1}, x_t] means concatenation: the previous hidden state and the current input are joined into a single vector and passed through a linear layer followed by sigmoid (σ). The sigmoid squashes every output to (0, 1), producing the per-dimension forgetting coefficients. This gate is then applied to the old cell state by element-wise multiplication: the old memories that are deemed irrelevant get multiplied toward zero, while important ones are multiplied by values close to 1 and survive.

Forget gate — sigmoid output applied to old cell state c_{t−1} element-wise
🧠 Analogy — Reading a New Paragraph
When you finish reading a paragraph and begin a new one, you remember the main theme of the document (that context is useful) but you release the specific grammatical structure of the last sentence (you no longer need to remember what the subject was for subject-verb agreement). The forget gate does exactly this — it has learned to release domain-specific short-range information when it becomes irrelevant, while retaining broader long-range context. In a trained language model, the forget gate reliably fires (outputs ≈ 0) for grammatical-agreement dimensions when a sentence boundary is encountered.
04
The Input Gate — Deciding What to Write
Gate 2 of 3: how the network generates new candidate memories and selectively stores them
▾

After the forget gate has decided what to erase from the old cell state, the input gate decides what new information to write in. This is actually a two-step process: first, a candidate cell state c̃_t is computed — a vector of potential updates, using tanh so values are bounded between −1 and +1. Second, the input gate i_t decides, dimension by dimension, how much of that candidate to actually commit to long-term memory. The separation between "what could be useful" (c̃_t) and "what to actually write" (i_t) gives the network fine-grained control over its memory writes.

Input gate and candidate cell state — two equations, one write decision
$$\mathbf{i}_t = \sigma\!\left(\mathbf{W}_i\,[\mathbf{h}_{t-1},\,\mathbf{x}_t] + \mathbf{b}_i\right) \qquad \text{(how much to write — values in (0,1))}$$ $$\tilde{\mathbf{c}}_t = \tanh\!\left(\mathbf{W}_c\,[\mathbf{h}_{t-1},\,\mathbf{x}_t] + \mathbf{b}_c\right) \qquad \text{(what to write — values in (-1,1))}$$

Notice the structural parallel with the forget gate: both i_t and c̃_t take the same two inputs (h_{t−1} and x_t concatenated), but one uses sigmoid (to produce a 0–1 gate) and the other uses tanh (to produce bounded candidate values that can be positive or negative — representing "this information should be stored with positive sign" or "with negative sign").

Input gate — two parallel computations then element-wise multiply to produce the memory write
🧠 Analogy — Taking Notes While Reading
The candidate c̃_t is like everything you could potentially write in your notes about the current sentence. The input gate i_t is your judgment about what is actually important enough to write down. Even if a sentence contains a lot of information, you might decide that only one or two key facts are worth committing to your notes. The LSTM makes exactly this judgment for every dimension of its memory independently — some dimensions it writes strongly, others it barely touches, others it completely ignores for this particular input.
05
The Output Gate — Deciding What to Expose
Gate 3 of 3: filtering the cell state into the working memory hidden state h_t
▾

After the forget and input gates have updated the cell state c_t — the long-term memory now contains a revised summary of all important history — the output gate decides how much of that memory to expose as the hidden state h_t. The hidden state is what the network actually uses for making predictions and what gets passed to the next time step as working context. The cell state might contain a lot of stored information, but not all of it is relevant for the current prediction. The output gate selectively filters what to surface.

Output gate and final hidden state computation
$$\mathbf{o}_t = \sigma\!\left(\mathbf{W}_o\,[\mathbf{h}_{t-1},\,\mathbf{x}_t] + \mathbf{b}_o\right)$$ $$\mathbf{h}_t = \mathbf{o}_t \odot \tanh(\mathbf{c}_t)$$

The tanh applied to c_t re-squashes the cell state values into (−1, 1) — this is important because the cell state can grow large in magnitude over many steps of accumulation. Then the output gate o_t filters this squashed cell state dimension by dimension. Dimensions where o_t ≈ 1 are fully exposed in h_t; dimensions where o_t ≈ 0 remain in the cell state but do not affect the current output or the next cell's input.

🧠 Analogy — Long-Term Memory vs Working Memory
Your long-term memory contains everything you have ever learned — but your working memory only holds what is relevant right now. You know calculus, but when you are buying groceries you do not need it active in your working memory. The output gate is the selective retrieval mechanism: it reads from long-term storage (cell state) and loads only what is relevant into working memory (hidden state). When generating the verb of a sentence, the output gate might expose "what is the current subject number" from the cell state but suppress "what was the topic of the previous paragraph".
06
The Complete LSTM Cell — All Gates Together
All five equations, the full data-flow diagram, parameter count, and why the architecture works
▾

Now that we have built each gate individually, we can see the complete LSTM as a unified system. Five equations define the entire computation — three gate equations, one candidate, and two state updates. Read them in order and notice how each one feeds into the next.

Complete LSTM equations — evaluated in this order at every time step t
$$\mathbf{f}_t = \sigma(\mathbf{W}_f[\mathbf{h}_{t-1},\mathbf{x}_t] + \mathbf{b}_f) \quad \text{(1) Forget gate — what to erase}$$ $$\mathbf{i}_t = \sigma(\mathbf{W}_i[\mathbf{h}_{t-1},\mathbf{x}_t] + \mathbf{b}_i) \quad \text{(2) Input gate — how much to write}$$ $$\tilde{\mathbf{c}}_t = \tanh(\mathbf{W}_c[\mathbf{h}_{t-1},\mathbf{x}_t] + \mathbf{b}_c) \quad \text{(3) Candidate — what to write}$$ $$\mathbf{c}_t = \mathbf{f}_t \odot \mathbf{c}_{t-1} + \mathbf{i}_t \odot \tilde{\mathbf{c}}_t \quad \text{(4) Cell state update — long-term memory}$$ $$\mathbf{o}_t = \sigma(\mathbf{W}_o[\mathbf{h}_{t-1},\mathbf{x}_t] + \mathbf{b}_o) \quad \text{(5) Output gate — what to expose}$$ $$\mathbf{h}_t = \mathbf{o}_t \odot \tanh(\mathbf{c}_t) \quad \text{(6) Hidden state — working memory}$$

The Complete Cell Diagram

Full LSTM cell — cell state flows horizontally across the top, gates operate on it from below

Parameter Count — Four Times a Vanilla RNN

An LSTM has four sets of weights — one for each of the three gates plus one for the candidate values — all sharing the same input structure [h_{t−1}, x_t]. If the input size is n and hidden size is h, each weight matrix is of shape (h, n+h), and there are 4 such matrices plus 4 bias vectors:

Total LSTM parameters
$$\text{Params} = 4 \times \underbrace{h \times (n + h)}_{\text{weight matrix}} + 4 \times \underbrace{h}_{\text{bias}} = 4h(n + h) + 4h$$

In practice this is often written as 4h(n + h + 1). An LSTM with input size 64 and hidden size 256 has 4 × 256 × (64 + 256 + 1) = 327,680 parameters — exactly 4× what a vanilla RNN of the same size would have. This is a meaningful but very reasonable cost for the enormous improvement in long-range memory capability.

✅ Why the Architecture Actually Works — Intuition
The three gates together implement something remarkably like selective attention to your own memory. The forget gate says "these dimensions of my stored knowledge are no longer relevant — clear them." The input gate says "these dimensions of new information are important enough to store." The output gate says "to make this particular prediction, I need to look at these aspects of my stored memory." Each gate is fully learned — the network discovers what to forget, write, and read entirely from the training signal, without any human guidance. This learned selectivity is what allows LSTMs to remember specific facts for hundreds of time steps while continuously accepting new information without overwriting everything.
07
Gated Recurrent Unit (GRU) — A Streamlined Alternative
Cho et al. (2014): merging cell and hidden state, collapsing to two gates, matching LSTM performance with fewer parameters
▾

The GRU, introduced by Cho et al. in 2014, asks a natural question: does an LSTM really need all that complexity? Three separate gate matrices, two separate state vectors, six separate equations — is all of it necessary? Their answer was that a great deal of it can be unified without sacrificing much expressive power. The GRU merges the cell state and hidden state into a single vector, collapses the forget and input gates into a single complementary pair, and eliminates the output gate entirely. The result is a model with two-thirds the parameters of an LSTM that often matches it in practice.

The Two GRU Gates

GRU has two gates: the reset gate r_t, which controls how much of the previous hidden state to use when computing candidate values (effectively deciding how much past context to "forget" when forming new memories), and the update gate z_t, which controls the blend between old hidden state and new candidate — a single gate doing the combined job of LSTM's forget and input gates.

Complete GRU equations — just three equations vs LSTM's six
$$\mathbf{z}_t = \sigma(\mathbf{W}_z[\mathbf{h}_{t-1},\,\mathbf{x}_t] + \mathbf{b}_z) \qquad \text{(1) Update gate — how much to update}$$ $$\mathbf{r}_t = \sigma(\mathbf{W}_r[\mathbf{h}_{t-1},\,\mathbf{x}_t] + \mathbf{b}_r) \qquad \text{(2) Reset gate — how much past to use}$$ $$\tilde{\mathbf{h}}_t = \tanh(\mathbf{W}_h[\mathbf{r}_t \odot \mathbf{h}_{t-1},\,\mathbf{x}_t] + \mathbf{b}_h) \qquad \text{(3) Candidate hidden state}$$ $$\mathbf{h}_t = (1 - \mathbf{z}_t) \odot \mathbf{h}_{t-1} + \mathbf{z}_t \odot \tilde{\mathbf{h}}_t \qquad \text{(4) Final hidden state}$$

Equation 4 is the most elegant part of the GRU and worth pausing on. It is a linear interpolation between the old hidden state h_{t−1} and the new candidate h̃_t, controlled by z_t. When z_t = 0 for a particular dimension, h_t = h_{t−1} — that dimension is completely preserved, unchanged, just like a forget gate of 1 combined with an input gate of 0 in LSTM. When z_t = 1, h_t = h̃_t — the old value is completely replaced by the new candidate. The complementary structure (1 − z_t) and z_t ensures the weights always sum to 1, giving a clean interpolation. This is the key insight: the GRU realises that "forget old" and "accept new" are not independent decisions — they are opposite ends of the same dial.

GRU cell — single hidden state, two gates, complementary update equation

The Reset Gate — What It Actually Does

The reset gate r_t warrants a closer look because its role is subtler than the update gate. When r_t = 1 (no reset), the candidate h̃_t is computed using the full previous hidden state — the standard recurrent behaviour. When r_t = 0 for certain dimensions, those dimensions of h_{t−1} are zeroed out before being fed into the candidate computation. This means the candidate is computed as if the network had no memory of those specific past states — effectively giving the candidate a "fresh start" for those dimensions. This is what allows GRU to model short-term patterns without the long-term memory interfering: when a new, locally-consistent sub-sequence begins, the reset gate suppresses the long-range history and lets the network respond to just the local context.

✅ GRU's Elegant Complementarity
The update gate equation h_t = (1−z_t)⊙h_{t-1} + z_t⊙h̃_t is beautiful in its simplicity. It is a weighted average of old and new, where the weights are constrained to sum to 1. This means the GRU cannot both strongly preserve old information AND strongly write new information to the same dimension simultaneously — it must choose. This constraint actually helps regularise the model: it cannot keep growing its hidden state magnitude without bound the way an unconstrained LSTM theoretically could. In practice this makes GRU slightly easier to train stably than LSTM.
08
LSTM vs GRU — When to Use Which
Architecture differences, empirical trade-offs, and practical decision rules
▾

The honest answer from over a decade of empirical research is: there is no universal winner. LSTM and GRU perform comparably on most tasks, and the difference is rarely large enough to be the deciding factor in a real project. But understanding their structural differences gives you principled intuitions for when to prefer one over the other.

Property LSTM GRU
State vectors Two: cell state cₜ (long-term) + hidden state hₜ (working) One: hidden state hₜ serves both roles
Number of gates 3 (forget, input, output) 2 (reset, update)
Parameter count 4h(n+h+1) 3h(n+h+1) — 25% fewer
Memory mechanism Explicit separate cell state — dedicated long-term storage that can be maintained independently of working memory Implicit — the single hidden state must serve as both long-term storage and working output
Training speed Slower per step (more parameters, more operations) Faster per step — often 20–30% faster wall-clock time on same hardware
Very long sequences Marginally better — the explicit cell state is a more direct gradient highway Slightly weaker — single state means memory and working signal compete for the same dimensions
Small datasets Prone to overfitting due to more parameters Often better — fewer parameters regularise the model naturally
Empirical winner Tends to win on large-scale NLP (machine translation, language modelling) with enough data Tends to match LSTM on smaller tasks; preferred in embedded/resource-constrained settings

When to prefer LSTM

Choose LSTM when your sequences are very long (hundreds of steps or more) and long-range dependencies are critical — for instance, resolving pronoun coreference across a paragraph, or modelling long musical structures. Also prefer LSTM when you have a large dataset and can afford the extra parameters, and when you are building on top of a large pretrained model where the LSTM weight structure is already established. The explicit separation of cell state and hidden state gives LSTM a slightly cleaner gradient path for truly long-range signals.

When to prefer GRU

Choose GRU when training speed matters (the 25% parameter reduction adds up on large datasets), when your dataset is small or moderately sized (fewer parameters reduce overfitting risk), when you are deploying on resource-constrained hardware like mobile or embedded systems, or when your sequences are moderate length and you have no strong reason to believe LSTM's extra machinery will help. GRU is also slightly easier to tune because there are fewer hyperparameter interactions.

⚠️ Practical Advice — Just Try Both
On any new task, the empirical answer almost always wins over the theoretical one. Run both LSTM and GRU with the same hidden size, same learning rate schedule, same dropout, and compare validation performance after the same number of training steps. In most cases the difference will be small enough that the faster-training GRU is the practical winner. If LSTM clearly wins, the task likely has very long-range dependencies worth the cost. The one case where you should default to LSTM without a comparison is when implementing from scratch to understand the architecture — the explicit cell state makes the gradient flow argument far cleaner to see.

A Third Perspective — Structural Comparison

It helps to see the two architectures side by side as pure data-flow patterns. The critical structural difference is that LSTM maintains an information pathway (the cell state) that flows through the network with only additive modifications — meaning it can pass gradient signal backward through hundreds of steps with minimal decay. GRU merges this into the hidden state, which is cleaner but means the gradient does pass through the update-gate multiplication at every step, albeit only element-wise rather than through a full weight matrix.

PyTorch Implementation — LSTM and GRU, From Scratch and with nn.LSTM
Building an LSTM cell manually to see every gate, then using the production API correctly
▾

As with vanilla RNNs, the best way to solidify the LSTM equations is to implement them from scratch in PyTorch before switching to nn.LSTM. When you write out each gate by hand you cannot gloss over any detail — every weight matrix, every sigmoid, every element-wise multiply becomes concrete. After that, the nn.LSTM API will feel like a transparent abstraction rather than a black box.

Part 1 — LSTM Cell From Scratch

Python · LSTM cell built from scratch — all 5 equations explicit
import torch
import torch.nn as nn

class LSTMCellScratch(nn.Module):
    """A single LSTM cell implementing all 6 equations explicitly.
    
    Inputs:  x_t  (batch, input_size)   — current input
             h_prev (batch, hidden_size) — previous hidden state
             c_prev (batch, hidden_size) — previous cell state
    Outputs: h_t  (batch, hidden_size)  — new hidden state (working memory)
             c_t  (batch, hidden_size)  — new cell state   (long-term memory)
    """
    def __init__(self, input_size: int, hidden_size: int):
        super().__init__()
        self.hidden_size = hidden_size

        # All four weight matrices take the same concatenated input [h_{t-1}, x_t]
        # PyTorch convention: combined W has shape (4*hidden, input+hidden)
        # We keep them separate here for clarity.
        n = input_size + hidden_size   # concatenated input size
        self.W_f = nn.Linear(n, hidden_size)  # forget gate
        self.W_i = nn.Linear(n, hidden_size)  # input gate
        self.W_c = nn.Linear(n, hidden_size)  # candidate cell state
        self.W_o = nn.Linear(n, hidden_size)  # output gate

    def forward(self, x_t, h_prev, c_prev):
        # Concatenate [h_{t-1}, x_t] → (batch, input+hidden)
        # All four gates share this same combined input
        combined = torch.cat([h_prev, x_t], dim=-1)

        # ── Gate computations ──────────────────────────────────────────
        f_t = torch.sigmoid(self.W_f(combined))  # forget: what % to keep from c_{t-1}
        i_t = torch.sigmoid(self.W_i(combined))  # input:  what % of candidate to write
        g_t = torch.tanh   (self.W_c(combined))  # candidate: new info (−1 to +1)
        o_t = torch.sigmoid(self.W_o(combined))  # output: what % to expose as h_t

        # ── State updates ──────────────────────────────────────────────
        # Cell state: keep some of old + add some of new candidate
        # This addition is the gradient highway — no weight matrix here!
        c_t = f_t * c_prev + i_t * g_t

        # Hidden state: filtered view of the cell state
        # tanh re-squashes c_t to (-1,1); output gate selects what to expose
        h_t = o_t * torch.tanh(c_t)

        return h_t, c_t


class LSTMScratch(nn.Module):
    """Full sequence LSTM using LSTMCellScratch with explicit time loop."""
    def __init__(self, input_size, hidden_size, output_size):
        super().__init__()
        self.hidden_size = hidden_size
        self.cell = LSTMCellScratch(input_size, hidden_size)
        self.head = nn.Linear(hidden_size, output_size)

    def forward(self, x):
        """x: (batch, seq_len, input_size)"""
        batch, seq_len, _ = x.shape
        h = torch.zeros(batch, self.hidden_size, device=x.device)
        c = torch.zeros(batch, self.hidden_size, device=x.device)

        outputs = []
        for t in range(seq_len):
            h, c = self.cell.forward(x[:, t, :], h, c)
            outputs.append(self.head(h))

        return torch.stack(outputs, dim=1)  # (batch, seq_len, output_size)


# ── Verify: parameter count should be 4 × h(n+h+1) ─────────────
cell = LSTMCellScratch(input_size=64, hidden_size=128)
params = sum(p.numel() for p in cell.parameters())
print(f"Params: {params}")
# Expected: 4 × 128 × (64 + 128 + 1) = 98,816

Part 2 — Production LSTM with nn.LSTM

Python · nn.LSTM API — critical shape details and common pitfalls
import torch
import torch.nn as nn

# ── nn.LSTM vs nn.RNN: one important difference in outputs ────────
# nn.RNN returns: (output, h_n)
# nn.LSTM returns: (output, (h_n, c_n))  ← note the TUPLE for hidden state!

lstm = nn.LSTM(
    input_size   = 64,
    hidden_size  = 128,
    num_layers   = 2,
    batch_first  = True,   # always use this!
    dropout      = 0.3,
    bidirectional= False,
)

x   = torch.randn(32, 50, 64)          # (batch=32, seq_len=50, input=64)
h0  = torch.zeros(2, 32, 128)           # (num_layers, batch, hidden)
c0  = torch.zeros(2, 32, 128)           # (num_layers, batch, hidden)

out, (h_n, c_n) = lstm.forward(x, (h0, c0))
# out:  (batch, seq_len, hidden)     — ALL hidden states at every step
# h_n:  (num_layers, batch, hidden)  — FINAL hidden state per layer
# c_n:  (num_layers, batch, hidden)  — FINAL cell state per layer

print(out.shape)   # → (32, 50, 128)
print(h_n.shape)   # → (2, 32, 128)
print(c_n.shape)   # → (2, 32, 128)


# ── Sequence Classifier with LSTM ────────────────────────────────
class LSTMClassifier(nn.Module):
    """Many-to-One: takes a variable-length sequence, outputs one class label.
    
    Design choices explained:
    - Embedding: learned dense representation of discrete tokens
    - LSTM: captures temporal dependencies
    - Take h_n[-1]: the last layer's final hidden state = summary of whole sequence
    - Dropout: regularisation before the linear classification head
    """
    def __init__(self, vocab_size, embed_dim, hidden_size,
                 num_layers, num_classes, dropout=0.3):
        super().__init__()
        self.embed = nn.Embedding(vocab_size, embed_dim, padding_idx=0)
        self.lstm  = nn.LSTM(embed_dim, hidden_size,
                              num_layers=num_layers, batch_first=True,
                              dropout=dropout if num_layers > 1 else 0.0)
        self.drop  = nn.Dropout(dropout)
        self.fc    = nn.Linear(hidden_size, num_classes)

    def forward(self, token_ids):
        """token_ids: (batch, seq_len) → logits: (batch, num_classes)"""
        x = self.embed(token_ids)              # (batch, seq, embed_dim)
        _, (h_n, _) = self.lstm(x)             # h_n: (layers, batch, hidden)
        h_last = self.drop(h_n[-1])            # last layer's final state
        return self.fc(h_last)                  # (batch, num_classes)


# ── Sequence Tagger with Bidirectional LSTM ───────────────────────
class BiLSTMTagger(nn.Module):
    """Many-to-Many: label every token (e.g. NER, POS tagging).
    
    Bidirectional: each position gets context from BOTH directions.
    hidden_size * 2 because forward and backward states are concatenated.
    """
    def __init__(self, vocab_size, embed_dim, hidden_size,
                 num_layers, num_tags, dropout=0.3):
        super().__init__()
        self.embed = nn.Embedding(vocab_size, embed_dim, padding_idx=0)
        self.lstm  = nn.LSTM(embed_dim, hidden_size,
                              num_layers=num_layers, batch_first=True,
                              bidirectional=True,
                              dropout=dropout if num_layers > 1 else 0.0)
        self.drop  = nn.Dropout(dropout)
        # hidden_size * 2 because bidirectional concatenates fwd + bwd
        self.fc    = nn.Linear(hidden_size * 2, num_tags)

    def forward(self, token_ids):
        """token_ids: (batch, seq_len) → tag_logits: (batch, seq_len, num_tags)"""
        x   = self.embed(token_ids)         # (batch, seq, embed)
        out, _ = self.lstm(x)               # out: (batch, seq, hidden*2)
        out = self.drop(out)
        return self.fc(out)                  # (batch, seq, num_tags)

Handling Variable-Length Sequences with Padding

In real datasets, sequences have different lengths. The standard approach is to pad shorter sequences to the length of the longest sequence in the batch with a special padding token (index 0), and then tell PyTorch to ignore those padded positions during the LSTM computation. This uses pack_padded_sequence and pad_packed_sequence — a pair of utilities that compress out the padding before feeding to the LSTM and restore the full padded shape afterward.

Python · Variable-length sequences — packing and unpacking
from torch.nn.utils.rnn import pack_padded_sequence, pad_packed_sequence

class LSTMWithPacking(nn.Module):
    def __init__(self, vocab_size, embed_dim, hidden_size, num_classes):
        super().__init__()
        self.embed = nn.Embedding(vocab_size, embed_dim, padding_idx=0)
        self.lstm  = nn.LSTM(embed_dim, hidden_size, batch_first=True)
        self.fc    = nn.Linear(hidden_size, num_classes)

    def forward(self, token_ids, lengths):
        """
        token_ids: (batch, max_len) — padded to the same length
        lengths:   (batch,)         — actual length of each sequence
        """
        x = self.embed(token_ids)            # (batch, max_len, embed_dim)

        # Pack: removes padding, tells LSTM where each sequence ends.
        # enforce_sorted=False: allows batches in any order
        packed = pack_padded_sequence(x, lengths.cpu(),
                                       batch_first=True,
                                       enforce_sorted=False)

        packed_out, (h_n, _) = self.lstm(packed)

        # Unpack: restores padded tensor (needed if you want per-step outputs)
        # out shape: (batch, max_len, hidden) — padded positions are 0
        out, _ = pad_packed_sequence(packed_out, batch_first=True)

        # For classification, just use the final hidden state of each sequence
        return self.fc(h_n[-1])               # (batch, num_classes)
🔑 The One API Gotcha — LSTM Returns a Tuple, RNN/GRU Does Not
The single most common mistake when switching between nn.RNN, nn.LSTM, and nn.GRU is forgetting that LSTM returns a tuple (h_n, c_n) for the hidden state, while RNN and GRU return just h_n. The correct unpacking is out, (h_n, c_n) = lstm(x) for LSTM, but out, h_n = gru(x) for GRU. Getting this wrong gives a cryptic shape error that is confusing until you know the cause. Always double-check the output signature when switching between recurrent layer types.

Application 1 — Sentiment Analysis (Many-to-One)

Sentiment analysis is the canonical LSTM classification task: read a review or tweet of arbitrary length and output a single label (positive/negative/neutral). The LSTM accumulates context as it reads each word — negation ("not great" differs from "great"), intensifiers ("absolutely terrible"), and long-range qualifications ("despite the terrible service, the food was outstanding") all require remembering earlier tokens when processing later ones. Vanilla RNNs fail on long reviews precisely because they cannot carry the negation context far enough forward; LSTMs handle this naturally.

Python · Sentiment analysis — complete training-ready LSTM pipeline
import torch, torch.nn as nn
from torch.nn.utils.rnn import pack_padded_sequence

class SentimentLSTM(nn.Module):
    """
    BiLSTM sentiment classifier with pretrained embedding option.
    Using bidirectional so both "not" (early) and the adjective (later)
    can inform each other — crucial for negation handling.
    """
    def __init__(self, vocab_size, embed_dim=128, hidden_size=256,
                 num_layers=2, num_classes=3, dropout=0.4,
                 pretrained_embeddings=None):
        super().__init__()

        self.embed = nn.Embedding(vocab_size, embed_dim, padding_idx=0)
        if pretrained_embeddings is not None:
            # Load GloVe or Word2Vec weights; freeze or fine-tune
            self.embed.weight.data.copy_(pretrained_embeddings)
            self.embed.weight.requires_grad = True  # fine-tune

        self.lstm = nn.LSTM(
            embed_dim, hidden_size, num_layers=num_layers,
            batch_first=True, bidirectional=True,
            dropout=dropout if num_layers > 1 else 0.0
        )
        self.drop = nn.Dropout(dropout)
        # hidden_size * 2: bidirectional concatenation
        self.fc   = nn.Linear(hidden_size * 2, num_classes)

    def forward(self, token_ids, lengths):
        x = self.embed(token_ids)
        packed = pack_padded_sequence(x, lengths.cpu(),
                                       batch_first=True, enforce_sorted=False)
        _, (h_n, _) = self.lstm(packed)
        # h_n shape: (num_layers * 2, batch, hidden) for bidirectional
        # h_n[-2]: forward direction final layer
        # h_n[-1]: backward direction final layer
        # Concatenate both directions for the full context representation
        h_cat = torch.cat([h_n[-2], h_n[-1]], dim=1)  # (batch, hidden*2)
        return self.fc(self.drop(h_cat))

# Training
model     = SentimentLSTM(vocab_size=30000)
optimizer = torch.optim.Adam(model.parameters(), lr=1e-3)
criterion = nn.CrossEntropyLoss()

Application 2 — Named Entity Recognition (Many-to-Many)

NER requires labelling every token in a sequence with a category — PERSON, ORGANISATION, LOCATION, DATE, or O (Other). This demands understanding context in both directions: the word "Washington" could be a person (George Washington), a place (Washington D.C.), or part of an organisation (Washington Post), and you often need to look at surrounding words to resolve this. A bidirectional LSTM is the standard architecture: the forward pass builds up left-context and the backward pass builds up right-context, and the concatenation of both hidden states at each position gives a rich, fully-contextualised representation for labelling.

Python · NER with BiLSTM — per-token classification with CRF option

    Standard architecture: embed → BiLSTM → dropout → linear per token
    For better boundary detection, a CRF layer can be added on top
    (not shown here, but torchcrf implements it cleanly).
    """
    def __init__(self, vocab_size, num_tags, embed_dim=100,
                 hidden_size=128, num_layers=1, dropout=0.5):
        super().__init__()
        self.embed = nn.Embedding(vocab_size, embed_dim, padding_idx=0)
        self.lstm  = nn.LSTM(embed_dim, hidden_size, num_layers=num_layers,
                              batch_first=True, bidirectional=True,
                              dropout=dropout if num_layers > 1 else 0.0)
        self.drop  = nn.Dropout(dropout)
        self.fc    = nn.Linear(hidden_size * 2, num_tags)

    def forward(self, token_ids):

The Remaining Limitation — Why Attention is Still Needed

LSTMs solve the vanishing gradient problem convincingly, but they do not solve everything. Two important limitations remain, and they directly motivate the next two pages of this module.

The fixed-size bottleneck. In a sequence-to-sequence architecture (encoder reads a source sentence, decoder generates a translation), the entire source sentence must be compressed into the final hidden state h_T before decoding begins. For short sentences this is manageable, but for long or complex inputs — a paragraph, a document, a long audio recording — a fixed-size vector cannot hold enough information. The beginning of a long source sentence is systematically under-represented in h_T because the LSTM has processed hundreds of subsequent tokens since then. Translation quality degrades noticeably on sentences longer than ~30 words with a vanilla encoder-decoder.

Sequential computation. Even with LSTMs, you cannot parallelise across time steps — step t+1 cannot begin until step t finishes, because h_t feeds into h_{t+1}. On modern GPU hardware with thousands of parallel cores, this sequential bottleneck severely limits throughput. Training an LSTM on a long document is hardware-inefficient in a deep sense — you are using a tiny fraction of available parallelism.
